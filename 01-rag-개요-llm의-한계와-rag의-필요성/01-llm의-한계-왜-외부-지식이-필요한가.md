# LLM의 한계 — 왜 외부 지식이 필요한가

> 대규모 언어 모델이 아무리 똑똑해 보여도, 혼자서는 절대 해결할 수 없는 근본적인 한계가 있습니다.

## 개요

이 섹션에서는 LLM(Large Language Model)이 가진 구조적 한계를 깊이 있게 분석합니다. LLM이 왜 거짓 정보를 자신 있게 말하는지, 왜 어제 일어난 사건을 모르는지, 왜 우리 회사의 내부 문서를 이해하지 못하는지 — 이 모든 질문에 대한 답을 찾아보겠습니다.

**선수 지식**: Python 기초 문법, API 호출 경험이 있으면 좋지만 없어도 괜찮습니다.
**학습 목표**:
- LLM의 파라메트릭 메모리(Parametric Memory) 개념을 이해한다
- 할루시네이션(Hallucination)이 왜 발생하는지 그 원인을 설명할 수 있다
- 지식 단절(Knowledge Cutoff)과 도메인 특화 부족 문제를 구분할 수 있다
- 이러한 한계를 극복하기 위한 접근법의 필요성을 납득한다

## 왜 알아야 할까?

2024년, 한 미국 변호사가 법정에 제출한 서류에서 ChatGPT가 생성한 판례를 인용했습니다. 문제는 그 판례가 **실제로 존재하지 않았다**는 것이죠. LLM이 자신 있게 만들어낸 가짜 판례명, 가짜 사건번호, 가짜 판결 요약이 법정 문서에 그대로 실렸던 겁니다.

이건 특수한 사례가 아닙니다. LLM을 프로덕션에 도입하려는 모든 개발자가 반드시 마주치는 문제입니다. "AI가 대답을 잘하는데 왜 RAG가 필요하지?"라고 생각한다면, 이 섹션을 통해 LLM의 **구조적 한계**를 정확히 이해해야 합니다. 한계를 알아야 올바른 해결책을 설계할 수 있으니까요.

## 핵심 개념

### 개념 1: 파라메트릭 메모리 — LLM의 "뇌"는 어떻게 작동하는가

> 💡 **비유**: LLM을 수천 권의 책을 읽고 시험을 본 학생이라고 상상해보세요. 이 학생은 책 내용을 **자기 머릿속에 요약·압축**해서 저장했습니다. 시험 중에 원본 책을 펼쳐볼 수는 없고, 오직 기억에 의존해서 답을 씁니다. 기억이 정확하면 훌륭한 답안이 나오지만, 기억이 흐릿하면? 그럴듯하게 지어내서 빈칸을 채우게 됩니다.

LLM의 지식은 **파라메트릭 메모리(Parametric Memory)**에 저장됩니다. "파라메트릭"이란 모델의 **파라미터(가중치)**에 인코딩되어 있다는 뜻입니다. GPT-4는 약 1.8조 개, LLaMA 3는 최대 4050억 개의 파라미터를 가지고 있는데, 학습 데이터의 패턴과 지식이 이 숫자들 속에 녹아들어 있는 거죠.

핵심적인 문제는 이겁니다: **학습이 끝나면 이 메모리는 동결(frozen)됩니다.** 새로운 정보를 추가하려면 모델 전체를 다시 학습시키거나 파인튜닝해야 하는데, 이는 엄청난 비용과 시간이 듭니다.

```python
# 파라메트릭 메모리의 개념을 코드로 비유해봅시다
class ParametricMemory:
    """LLM의 파라메트릭 메모리를 단순화한 비유"""

    def __init__(self, training_data: list[str]):
        # 학습 시: 데이터를 파라미터(가중치)로 압축 저장
        self.parameters = self._compress(training_data)
        self.training_cutoff = "2024-04-01"  # 학습 데이터의 마지막 날짜

    def _compress(self, data: list[str]) -> dict:
        # 원본 데이터는 버리고, 패턴만 추출하여 저장
        # (실제로는 수십억 개의 부동소수점 숫자)
        return {"patterns": "compressed_knowledge", "size": "1.8T params"}

    def answer(self, question: str) -> str:
        # 추론 시: 파라미터에서 패턴을 꺼내 답변 생성
        # 원본 데이터를 참조할 수 없음!
        return self._generate_from_patterns(question)
```

이 구조가 만들어내는 세 가지 근본적 한계를 하나씩 살펴보겠습니다.

### 개념 2: 할루시네이션 — 자신 있게 거짓말하는 AI

> 💡 **비유**: 할루시네이션은 시험에서 답을 모를 때 "빈칸을 비워두는" 대신 **그럴듯하게 지어내는** 것과 같습니다. 문제는 이 학생이 너무 글을 잘 써서, 채점관도 속을 정도로 그럴듯한 가짜 답안을 만들어낸다는 거죠.

**할루시네이션(Hallucination)**이란 LLM이 학습 데이터에 없거나 사실이 아닌 정보를 마치 사실인 것처럼 자신 있게 생성하는 현상입니다. 왜 이런 일이 일어날까요?

2025년 9월, OpenAI 연구진(Kalai, Nachum, Vempala, Zhang)은 ["Why Language Models Hallucinate"](https://arxiv.org/abs/2509.04664)라는 논문을 발표했습니다. 이 논문의 핵심 주장은 명확합니다: **다음 토큰 예측(Next-Token Prediction)이라는 학습 방식 자체가 할루시네이션을 유발한다**는 것입니다.

LLM은 "주어진 텍스트 다음에 올 가장 가능성 높은 토큰"을 예측하도록 학습됩니다. 이 과정에서 **"모르겠습니다"라고 답하면 점수가 낮아지기 때문에**, 모델은 확신이 없어도 그럴듯한 답을 내놓는 방향으로 학습되는 거죠.

```run:python
# 할루시네이션이 발생하는 구조적 원인을 시뮬레이션
import random

def next_token_prediction(context: str, knowledge_base: dict) -> str:
    """다음 토큰 예측 방식의 단순화된 시뮬레이션"""
    # LLM은 확률 분포에서 다음 토큰을 선택
    if context in knowledge_base:
        return knowledge_base[context]  # 학습 데이터에 있으면 정확한 답
    else:
        # 학습 데이터에 없으면? "모르겠다" 대신 패턴 기반으로 생성!
        plausible_answers = ["그럴듯한 가짜 답변 A", "그럴듯한 가짜 답변 B"]
        return random.choice(plausible_answers)  # ← 할루시네이션 발생!

# 학습 데이터에 있는 질문
knowledge = {"파이썬 창시자는?": "귀도 반 로섬(Guido van Rossum)"}

print("=== 학습 데이터에 있는 질문 ===")
print(f"Q: 파이썬 창시자는?")
print(f"A: {next_token_prediction('파이썬 창시자는?', knowledge)}")

print("\n=== 학습 데이터에 없는 질문 ===")
print(f"Q: 2026년 1월 출시된 신규 프레임워크는?")
print(f"A: {next_token_prediction('2026년 1월 출시된 신규 프레임워크는?', knowledge)}")
print("   ↑ 모르는 건 지어냅니다 (할루시네이션)")
```

```output
=== 학습 데이터에 있는 질문 ===
Q: 파이썬 창시자는?
A: 귀도 반 로섬(Guido van Rossum)

=== 학습 데이터에 없는 질문 ===
Q: 2026년 1월 출시된 신규 프레임워크는?
A: 그럴듯한 가짜 답변 A
   ↑ 모르는 건 지어냅니다 (할루시네이션)
```

Vectara의 [할루시네이션 리더보드](https://huggingface.co/spaces/vectara/leaderboard)에 따르면, 최신 모델들도 여전히 할루시네이션에서 자유롭지 않습니다. 단순 문서 요약 작업에서조차 사실과 다른 내용을 추가하는 비율이 측정되고 있죠. 모델이 발전할수록 할루시네이션의 빈도는 줄어들고 있지만, **구조적으로 완전히 제거하는 것은 불가능**하다는 것이 학계의 중론입니다.

> ⚠️ **흔한 오해**: "최신 모델은 할루시네이션을 거의 안 한다"고 생각하기 쉽지만, 실제로는 평가 방법에 따라 할루시네이션 비율이 크게 달라집니다. 단순한 요약 작업에서는 낮은 비율을 보이지만, 전문 도메인이나 최신 정보를 묻는 질문에서는 여전히 상당한 비율로 발생합니다. 특히 OpenAI의 o3 모델은 PersonQA 벤치마크에서 전작 대비 오히려 할루시네이션이 증가한 사례도 보고되었습니다.

### 개념 3: 지식 단절 — "어제 무슨 일이 있었는지 모르겠어요"

> 💡 **비유**: 지식 단절은 **2024년 4월에 동결된 백과사전**을 사용하는 것과 같습니다. 그 이후에 일어난 모든 일 — 새로운 법률, 새로운 기술, 새로운 사건 — 은 이 백과사전에 없습니다. 그런데 이 백과사전은 "해당 항목이 없습니다"라고 말하는 대신, 기존 정보를 조합해서 그럴듯한 답을 지어냅니다.

**지식 단절(Knowledge Cutoff)**은 LLM 학습 데이터의 마지막 시점을 의미합니다. 이 시점 이후의 정보는 모델이 전혀 알지 못합니다.

| 모델 | 지식 단절 시점 |
|------|--------------|
| GPT-4o | 2024년 10월 |
| GPT-4 Turbo | 2023년 12월 |
| Claude 3.5 Sonnet | 2024년 4월 |
| Claude Sonnet 4 | 2025년 3월 |
| LLaMA 3 | 2023년 12월 |

실무에서 이것이 왜 위험한지 구체적인 예를 들어보겠습니다.

```run:python
# 지식 단절이 실무에서 만드는 문제 시나리오
from dataclasses import dataclass
from datetime import date

@dataclass
class LLMKnowledge:
    """LLM이 알고 있는 지식의 범위"""
    cutoff_date: date
    known_facts: dict[str, str]

# 2024년 4월까지의 데이터로 학습된 모델
model = LLMKnowledge(
    cutoff_date=date(2024, 4, 1),
    known_facts={
        "최신 Python 버전": "3.12",        # 학습 시점 기준
        "LangChain 최신 버전": "0.1.x",    # 학습 시점 기준
        "회사 내부 보안 정책": "???",        # 애초에 학습 데이터에 없음
    }
)

# 2026년 3월에 사용자가 질문하는 상황
questions = [
    ("최신 Python 버전은?", "3.13 (2024년 10월 출시)", "3.12라고 답할 가능성 높음"),
    ("LangChain 최신 버전은?", "1.0.x (대규모 변경)", "0.1.x 기준으로 답변 → 코드 오류 유발"),
    ("우리 회사 휴가 정책은?", "회사 내부 문서에만 존재", "모르면서 일반적인 정책을 지어냄"),
]

print("=" * 65)
print(f"모델 지식 단절: {model.cutoff_date}")
print(f"현재 날짜: 2026-03-03")
print("=" * 65)

for q, reality, risk in questions:
    print(f"\n📌 질문: {q}")
    print(f"   현실: {reality}")
    print(f"   ⚠️  위험: {risk}")
```

```output
=================================================================
모델 지식 단절: 2024-04-01
현재 날짜: 2026-03-03
=================================================================

📌 질문: 최신 Python 버전은?
   현실: 3.13 (2024년 10월 출시)
   ⚠️  위험: 3.12라고 답할 가능성 높음

📌 질문: LangChain 최신 버전은?
   현실: 1.0.x (대규모 변경)
   ⚠️  위험: 0.1.x 기준으로 답변 → 코드 오류 유발

📌 질문: 우리 회사 휴가 정책은?
   현실: 회사 내부 문서에만 존재
   ⚠️  위험: 모르면서 일반적인 정책을 지어냄
```

특히 주목할 점은 **실효 단절 시점(Effective Cutoff)**이 공식 발표와 다를 수 있다는 것입니다. 한 연구("Dated Data: Tracing Knowledge Cutoffs in Large Language Models", 2024)에 따르면, LLM의 학습 데이터는 모든 주제에 대해 균일하게 단절되지 않습니다. 예를 들어 유명 인물에 대한 정보는 비교적 최신까지 포함되어 있지만, 특정 기술 문서나 법률 정보는 공식 단절 시점보다 훨씬 이전 버전만 학습되어 있을 수 있죠.

### 개념 4: 도메인 특화 부족 — "일반 상식은 있지만 전문가는 아닙니다"

> 💡 **비유**: LLM은 **모든 과목을 두루 공부한 제너럴리스트**입니다. 의학, 법률, 금융, 공학 어떤 질문에도 답할 수 있지만, 심장외과 전문의처럼 특정 수술의 미세한 차이를 알거나, 세무사처럼 올해 바뀐 세법을 정확히 적용하지는 못합니다.

LLM의 학습 데이터는 인터넷에 공개된 텍스트가 주를 이룹니다. 이 말은 곧:

1. **기업 내부 문서**는 학습 데이터에 포함되지 않습니다 — 사내 위키, 내부 API 문서, 고객 데이터 등
2. **최신 전문 지식**은 반영되지 않습니다 — 최근 발표된 의학 논문, 새로 개정된 법률, 업데이트된 기술 스펙
3. **특수 도메인 용어**를 정확히 이해하지 못합니다 — 산업별 전문 용어, 기업 고유 약어

```run:python
# 도메인 특화 부족 문제 시나리오
scenarios = {
    "의료": {
        "질문": "환자에게 최신 항암제 X를 처방해도 될까?",
        "LLM 문제": "학습 시점 이후 승인된 약물 정보 부재, 최신 임상시험 결과 미반영",
        "위험도": "🔴 높음 (환자 안전 직결)",
    },
    "법률": {
        "질문": "2025년 5월 개정된 노동법에 따른 퇴직금 계산은?",
        "LLM 문제": "개정 전 법률 기준으로 답변, 잘못된 계산식 제공",
        "위험도": "🔴 높음 (법적 분쟁 가능)",
    },
    "기업 내부": {
        "질문": "우리 회사의 API v3 인증 방식은?",
        "LLM 문제": "내부 문서에만 있는 정보, 일반적인 인증 방식을 추측으로 답변",
        "위험도": "🟡 중간 (시스템 오류 가능)",
    },
}

for domain, info in scenarios.items():
    print(f"📋 [{domain}] {info['질문']}")
    print(f"   문제: {info['LLM 문제']}")
    print(f"   위험도: {info['위험도']}")
    print()
```

```output
📋 [의료] 환자에게 최신 항암제 X를 처방해도 될까?
   문제: 학습 시점 이후 승인된 약물 정보 부재, 최신 임상시험 결과 미반영
   위험도: 🔴 높음 (환자 안전 직결)

📋 [법률] 2025년 5월 개정된 노동법에 따른 퇴직금 계산은?
   문제: 개정 전 법률 기준으로 답변, 잘못된 계산식 제공
   위험도: 🔴 높음 (법적 분쟁 가능)

📋 [기업 내부] 우리 회사의 API v3 인증 방식은?
   문제: 내부 문서에만 있는 정보, 일반적인 인증 방식을 추측으로 답변
   위험도: 🟡 중간 (시스템 오류 가능)
```

이 세 가지 한계를 정리하면, LLM은 **"과거에 공개적으로 존재하던 일반 지식"**에는 강하지만, **"최신의, 비공개의, 전문적인 지식"**에는 구조적으로 취약합니다. 그리고 더 큰 문제는 — LLM 스스로가 이 한계를 인식하지 못한다는 것입니다.

## 실습: 직접 해보기

LLM의 한계를 직접 체험해봅시다. 아래 코드는 OpenAI API를 사용하여 LLM에게 다양한 유형의 질문을 던지고, 각 응답에서 한계를 분석합니다.

```python
# 실습: LLM의 한계를 직접 확인하기
# 필요 패키지: pip install openai python-dotenv

import os
from openai import OpenAI
from dotenv import load_dotenv

load_dotenv()  # .env 파일에서 OPENAI_API_KEY 로드

client = OpenAI()

def ask_llm(question: str, system_prompt: str = "") -> str:
    """LLM에게 질문하고 응답을 받는 함수"""
    messages = []
    if system_prompt:
        messages.append({"role": "system", "content": system_prompt})
    messages.append({"role": "user", "content": question})

    response = client.chat.completions.create(
        model="gpt-4o",
        messages=messages,
        temperature=0.0,  # 재현 가능하도록 temperature를 0으로 설정
        max_tokens=500,
    )
    return response.choices[0].message.content


# === 테스트 1: 할루시네이션 유도 ===
print("=" * 60)
print("테스트 1: 할루시네이션 (존재하지 않는 정보)")
print("=" * 60)
# 존재하지 않는 논문에 대해 질문
response = ask_llm(
    "Kim et al. (2025)의 'Neural Quantum RAG' 논문의 "
    "핵심 기여를 3가지로 요약해주세요."
)
print(f"Q: 존재하지 않는 논문에 대한 질문")
print(f"A: {response}")
print(f"⚠️  이 논문은 존재하지 않습니다. LLM이 지어낸 답변입니다!\n")


# === 테스트 2: 지식 단절 확인 ===
print("=" * 60)
print("테스트 2: 지식 단절 (Knowledge Cutoff)")
print("=" * 60)
response = ask_llm(
    "당신의 학습 데이터는 언제까지의 정보를 포함하고 있나요? "
    "그리고 오늘 날짜의 주요 뉴스를 알려주세요."
)
print(f"Q: 학습 데이터 범위와 최신 뉴스 질문")
print(f"A: {response}\n")


# === 테스트 3: 도메인 특화 지식 부재 ===
print("=" * 60)
print("테스트 3: 도메인 특화 지식 부재")
print("=" * 60)
response = ask_llm(
    "CourseKit 프레임워크의 pipeline.py에서 "
    "Phase 1.5 Chapter Review가 작동하는 방식을 설명해주세요."
)
print(f"Q: 비공개 프로젝트의 내부 구현 질문")
print(f"A: {response}")
print(f"⚠️  CourseKit은 내부 프로젝트입니다. LLM은 알 수 없습니다!\n")


# === 결과 분석 ===
print("=" * 60)
print("📊 분석 요약")
print("=" * 60)
print("1. 할루시네이션: 존재하지 않는 논문도 자신 있게 설명")
print("2. 지식 단절: 학습 이후 정보에 대해 답변 불가 또는 추측")
print("3. 도메인 특화: 내부/비공개 지식에 대해 일반론으로 대체")
print("\n→ 이 세 가지 한계가 RAG의 필요성을 만듭니다!")
```

> 🔥 **실무 팁**: 위 코드에서 `temperature=0.0`으로 설정한 이유가 있습니다. Temperature를 높이면 더 창의적인(= 더 많이 지어내는) 답변이 나옵니다. 프로덕션에서 사실 기반 답변이 필요할 때는 temperature를 낮게 설정하는 것이 기본이지만, 이것만으로는 할루시네이션을 막을 수 없습니다. 근본적인 해결책이 필요한 이유죠.

## 더 깊이 알아보기

### RAG의 탄생 배경: Lewis et al. (2020)

LLM의 한계를 가장 체계적으로 분석하고, 이에 대한 해결책을 제시한 논문이 바로 2020년 Meta AI(당시 Facebook AI Research)에서 발표한 ["Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks"](https://arxiv.org/abs/2005.11401)입니다.

이 논문의 주저자인 **Patrick Lewis**와 동료 연구자들은 흥미로운 관찰에서 출발했습니다. 당시 최고 성능의 언어 모델들도 "지식 집약적 작업(Knowledge-Intensive Tasks)"에서는 형편없는 성능을 보였거든요. 오픈 도메인 QA, 팩트 체크, 지식 기반 대화 같은 작업에서 말이죠.

Lewis 팀의 핵심 아이디어는 이랬습니다: **모든 지식을 모델 파라미터에 억지로 밀어넣는 대신, 필요할 때 외부에서 가져오면 어떨까?** 마치 시험을 볼 때 모든 걸 외우는 대신, 참고서를 옆에 두고 필요할 때 찾아보는 것처럼요.

그래서 그들은 두 가지 메모리를 결합한 모델을 제안했습니다:
- **파라메트릭 메모리**: 기존 LLM(BART)의 가중치에 저장된 지식
- **비파라메트릭 메모리**: 위키피디아 전체를 벡터로 인덱싱한 외부 검색기(DPR)

이것이 바로 **RAG(Retrieval-Augmented Generation)**, 검색 증강 생성의 시작이었습니다. 놀랍게도, 이 아이디어는 세 개의 오픈 도메인 QA 벤치마크에서 최고 성능을 달성했고, 기존의 파라메트릭 모델뿐 아니라 전용 검색-추출 아키텍처보다도 뛰어난 성능을 보였습니다.

### 왜 "할루시네이션은 불가피한가"

2024년 1월, ["Hallucination is Inevitable: An Innate Limitation of Large Language Models"](https://arxiv.org/abs/2401.11817)이라는 도발적인 제목의 논문이 발표되었습니다. 이 논문은 수학적으로 증명합니다 — LLM이 범용 문제 해결기로 사용되는 한, 할루시네이션은 **원리적으로 제거 불가능**하다는 것을요.

직관적으로 이해하면 이렇습니다: LLM은 학습 데이터의 **통계적 패턴**을 학습합니다. 하지만 세상의 모든 사실을 통계적 패턴만으로 표현할 수는 없죠. 특히 드물게 등장하는 사실(long-tail facts)은 패턴만으로는 정확히 재현할 수 없고, 결국 모델은 "가장 그럴듯한 답"을 생성하게 됩니다 — 그것이 사실이든 아니든요.

> 💡 **알고 계셨나요?**: "할루시네이션"이라는 용어는 원래 **신경과학과 심리학**에서 온 것입니다. 실제로 존재하지 않는 것을 지각하는 현상을 뜻하죠. AI 분야에서 이 용어를 차용한 것은, LLM이 마치 "존재하지 않는 정보를 본 것처럼" 자신 있게 생성하는 행동이 인간의 환각 현상과 유사하기 때문입니다. 다만, 인간의 환각은 뇌의 오작동이지만 LLM의 할루시네이션은 **정상 작동의 부산물**이라는 점이 다릅니다.

## 흔한 오해와 팁

> ⚠️ **흔한 오해**: "모델을 더 크게 만들면 할루시네이션이 사라진다." — 사실이 아닙니다. 모델 크기가 커지면 일반적인 사실에 대한 정확도는 높아지지만, 학습 데이터에 없는 정보를 지어내는 구조적 문제는 해결되지 않습니다. GPT-4가 GPT-3.5보다 정확하지만, 여전히 할루시네이션을 하는 것처럼요. 심지어 OpenAI의 o3 모델은 일부 벤치마크에서 전작보다 할루시네이션이 증가했습니다.

> 💡 **알고 계셨나요?**: LLM의 지식 단절이 모든 주제에 대해 균일하지 않다는 연구 결과가 있습니다. 유명인에 대한 정보는 비교적 최신까지 포함되어 있지만, 특정 기술 문서나 법률 정보는 공식 단절 시점보다 수개월~수년 전 버전만 학습되어 있을 수 있습니다. 이를 **실효 단절(Effective Cutoff)**이라 부릅니다.

> 🔥 **실무 팁**: LLM 기반 서비스를 만들 때, "이 정보가 LLM의 학습 데이터에 포함되어 있을까?"라고 의심하는 습관을 들이세요. 특히 다음 세 가지는 거의 확실히 LLM이 모르는 영역입니다: (1) 지난 6개월 이내의 정보, (2) 회사 내부 데이터, (3) 니치한 전문 도메인 지식. 이런 질문이 예상된다면, 처음부터 RAG를 설계에 포함해야 합니다.

## 핵심 정리

| 개념 | 설명 |
|------|------|
| 파라메트릭 메모리 | LLM의 지식이 모델 가중치(파라미터)에 압축 저장되는 방식. 학습 후 동결됨 |
| 할루시네이션 | LLM이 사실이 아닌 정보를 자신 있게 생성하는 현상. 다음 토큰 예측 구조의 부산물 |
| 지식 단절 | LLM 학습 데이터의 마지막 시점. 이후 정보는 알 수 없음 |
| 실효 단절 | 공식 단절 시점과 실제 지식 범위가 다를 수 있는 현상 |
| 도메인 특화 부족 | 내부 문서, 최신 전문 지식, 특수 용어를 다루지 못하는 한계 |
| 비파라메트릭 메모리 | RAG에서 외부 검색 시스템을 통해 접근하는 지식 저장소 |

## 다음 섹션 미리보기

LLM의 한계를 확인했으니, 이제 해결책을 알아볼 차례입니다. 다음 섹션 **"RAG란 무엇인가 — 검색 증강 생성의 핵심 아이디어"**에서는 Lewis et al.이 제안한 RAG의 구체적인 동작 원리를 파헤쳐 봅니다. "검색(Retrieval) → 증강(Augmentation) → 생성(Generation)"이라는 세 단계가 어떻게 LLM의 한계를 극복하는지, 그리고 파라메트릭 메모리와 비파라메트릭 메모리를 결합한다는 것이 실제로 어떤 의미인지 알아보겠습니다.
