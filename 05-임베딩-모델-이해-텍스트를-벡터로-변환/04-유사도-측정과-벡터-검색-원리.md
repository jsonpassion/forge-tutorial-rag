## 핵심 개념

### 개념 1: 코사인 유사도 — 방향만 본다

> 💡 **비유**: 두 사람이 서울역 광장에 서 있다고 상상해보세요. 한 명은 남산타워를 가리키고, 다른 한 명도 남산타워를 가리킵니다. 한 사람이 팔이 더 길다고(벡터의 크기가 크다고) 해서 가리키는 **방향**이 달라지진 않죠? 코사인 유사도는 바로 이 **"가리키는 방향이 얼마나 같은가"**만 측정합니다.

코사인 유사도(Cosine Similarity)는 두 벡터가 이루는 **각도의 코사인 값**입니다. 수식으로 표현하면:

$$\text{cos\_sim}(\mathbf{a}, \mathbf{b}) = \frac{\mathbf{a} \cdot \mathbf{b}}{|\mathbf{a}| \times |\mathbf{b}|} = \frac{\sum_{i=1}^{n} a_i b_i}{\sqrt{\sum_{i=1}^{n} a_i^2} \times \sqrt{\sum_{i=1}^{n} b_i^2}}$$

각 기호의 의미:
- $\mathbf{a} \cdot \mathbf{b}$: 두 벡터의 내적(dot product)
- $|\mathbf{a}|$: 벡터 $\mathbf{a}$의 크기(L2 노름)
- $n$: 벡터의 차원 수

이게 의미하는 바는, **분자**(내적)가 두 벡터의 방향과 크기를 모두 반영하지만, **분모**(크기의 곱)로 나눠줌으로써 크기의 영향을 상쇄한다는 겁니다. 결과값은 **-1에서 1** 사이이며:

| 값 | 의미 |
|-----|------|
| 1 | 완전히 같은 방향 (동일한 의미) |
| 0 | 직교 (관련 없음) |
| -1 | 완전히 반대 방향 |

> 실제 임베딩에서는 대부분 0~1 사이의 값이 나옵니다. 음수가 나오는 경우는 드물어요.

NumPy로 직접 구현해봅시다:

```run:python
import numpy as np

def cosine_similarity(a: np.ndarray, b: np.ndarray) -> float:
    """두 벡터의 코사인 유사도를 계산합니다."""
    dot_product = np.dot(a, b)          # 분자: 내적
    norm_a = np.linalg.norm(a)          # 벡터 a의 크기
    norm_b = np.linalg.norm(b)          # 벡터 b의 크기
    return dot_product / (norm_a * norm_b)  # 코사인 유사도

# 간단한 예제: 3차원 벡터
v1 = np.array([1.0, 2.0, 3.0])
v2 = np.array([2.0, 4.0, 6.0])   # v1과 같은 방향, 크기만 2배
v3 = np.array([3.0, -1.0, 0.5])  # v1과 다른 방향

print(f"v1과 v2 (같은 방향): {cosine_similarity(v1, v2):.4f}")
print(f"v1과 v3 (다른 방향): {cosine_similarity(v1, v3):.4f}")
```

```output
v1과 v2 (같은 방향): 1.0000
v1과 v3 (다른 방향): 0.3656
```

v1과 v2는 크기가 2배 차이나지만 방향이 같아 유사도가 1.0입니다. 코사인 유사도가 **크기를 무시하고 방향만 본다**는 것을 확인할 수 있죠.

### 개념 2: 유클리드 거리 — 직선 거리를 잰다

> 💡 **비유**: 지도에서 두 지점 사이에 자를 대고 **직선 거리**를 재는 것과 같습니다. 서울에서 부산까지의 직선 거리처럼요. 가까울수록 유사하고, 멀수록 다릅니다.

유클리드 거리(Euclidean Distance)는 벡터 공간에서 두 점 사이의 **직선 거리**입니다. 피타고라스 정리의 다차원 확장이라고 볼 수 있어요:

$$d(\mathbf{a}, \mathbf{b}) = \sqrt{\sum_{i=1}^{n} (a_i - b_i)^2} = \|\mathbf{a} - \mathbf{b}\|_2$$

각 기호의 의미:
- $(a_i - b_i)^2$: 각 차원에서의 차이를 제곱
- $\sqrt{}$: 제곱합의 제곱근 (L2 노름)

이게 의미하는 바는, 각 차원의 차이를 모두 고려하기 때문에 **벡터의 크기(magnitude)에 민감하다**는 것입니다. 결과값은 **0부터 무한대**까지이며:

| 값 | 의미 |
|------|------|
| 0 | 완전히 같은 벡터 |
| 작을수록 | 유사함 |
| 클수록 | 다름 |

> ⚠️ 주의: 유클리드 거리는 "거리"이므로 **작을수록 유사**합니다. 유사도(similarity)와는 반대 방향이라는 점에 주의하세요!

```run:python
import numpy as np

def euclidean_distance(a: np.ndarray, b: np.ndarray) -> float:
    """두 벡터 사이의 유클리드 거리를 계산합니다."""
    return np.linalg.norm(a - b)  # L2 노름 = 유클리드 거리

v1 = np.array([1.0, 2.0, 3.0])
v2 = np.array([2.0, 4.0, 6.0])   # v1과 같은 방향, 크기 2배
v3 = np.array([3.0, -1.0, 0.5])  # v1과 다른 방향

print(f"v1과 v2 (같은 방향, 크기 다름): {euclidean_distance(v1, v2):.4f}")
print(f"v1과 v3 (다른 방향):           {euclidean_distance(v1, v3):.4f}")
print(f"v1과 v1 (자기 자신):           {euclidean_distance(v1, v1):.4f}")
```

```output
v1과 v2 (같은 방향, 크기 다름): 3.7417
v1과 v3 (다른 방향):           4.3012
v1과 v1 (자기 자신):           0.0000
```

흥미로운 점을 보셨나요? 코사인 유사도에서 v1과 v2는 완벽하게 같았는데(1.0), 유클리드 거리에서는 3.74나 떨어져 있습니다. **크기 차이가 거리에 반영**되기 때문이에요.

### 개념 3: 내적(Dot Product) — 방향과 크기를 모두 본다

> 💡 **비유**: 두 사람이 밧줄을 같은 방향으로 당기고 있다면, 합쳐진 힘은 각자의 힘(크기)과 방향에 모두 달려 있죠. 같은 방향으로 세게 당길수록 합력이 커지듯, 내적도 **방향이 같고 크기가 클수록** 값이 커집니다.

내적(Dot Product)은 가장 기본적인 벡터 연산 중 하나입니다:

$$\mathbf{a} \cdot \mathbf{b} = \sum_{i=1}^{n} a_i b_i = |\mathbf{a}||\mathbf{b}|\cos\theta$$

각 기호의 의미:
- $a_i b_i$: 각 차원의 요소를 곱하여 합산
- $\theta$: 두 벡터 사이의 각도
- $|\mathbf{a}||\mathbf{b}|$: 두 벡터의 크기의 곱

이게 의미하는 바는, 내적이 **코사인 유사도의 분자**라는 것입니다. 코사인 유사도가 크기를 나눠서 제거하는 반면, 내적은 크기를 그대로 유지합니다. 그래서:

| 조건 | 결과 |
|------|------|
| 같은 방향 + 큰 크기 | 큰 양수 |
| 같은 방향 + 작은 크기 | 작은 양수 |
| 직교 | 0 |
| 반대 방향 | 음수 |

```run:python
import numpy as np

def dot_product(a: np.ndarray, b: np.ndarray) -> float:
    """두 벡터의 내적을 계산합니다."""
    return np.dot(a, b)

v1 = np.array([1.0, 2.0, 3.0])
v2 = np.array([2.0, 4.0, 6.0])   # v1과 같은 방향, 크기 2배
v3 = np.array([3.0, -1.0, 0.5])  # v1과 다른 방향

print(f"v1과 v2 (같은 방향, 크기 2배): {dot_product(v1, v2):.4f}")
print(f"v1과 v3 (다른 방향):          {dot_product(v1, v3):.4f}")
print(f"v1과 v1 (자기 자신):          {dot_product(v1, v1):.4f}")
```

```output
v1과 v2 (같은 방향, 크기 2배): 28.0000
v1과 v3 (다른 방향):          2.5000
v1과 v1 (자기 자신):          14.0000
```

v1·v2 = 28인데 v1·v1 = 14입니다. v2가 v1의 2배 크기이기 때문에 내적도 2배가 되었죠. 코사인 유사도에서는 둘 다 1.0이었지만, 내적은 **크기 정보를 보존**합니다.

### 개념 4: 세 메트릭의 관계 — 정규화의 마법

여기서 아주 중요한 사실이 있습니다. **벡터를 정규화(단위 벡터로 변환)하면, 세 메트릭은 본질적으로 같아집니다.**

정규화란 벡터의 크기를 1로 만드는 것입니다: $\hat{\mathbf{a}} = \frac{\mathbf{a}}{|\mathbf{a}|}$

정규화된 벡터에서는:
- **코사인 유사도 = 내적**: 분모가 $1 \times 1 = 1$이므로 코사인 유사도가 곧 내적
- **유클리드 거리와 코사인 유사도의 관계**: $d^2 = 2(1 - \text{cos\_sim})$

이것이 왜 중요할까요? OpenAI의 임베딩은 **이미 정규화되어 나옵니다**. 공식 문서에서도 "We recommend cosine similarity. The choice of distance function typically doesn't matter much"라고 말하는 이유가 바로 이것이에요. 정규화된 벡터에서는 어떤 메트릭을 써도 **검색 순위가 동일**하기 때문입니다.

```run:python
import numpy as np

# 벡터 정규화
def normalize(v: np.ndarray) -> np.ndarray:
    """벡터를 단위 벡터로 정규화합니다."""
    return v / np.linalg.norm(v)

v1 = np.array([1.0, 2.0, 3.0])
v3 = np.array([3.0, -1.0, 0.5])

# 정규화
v1_norm = normalize(v1)
v3_norm = normalize(v3)

# 정규화 후 세 메트릭 비교
cos_sim = np.dot(v1_norm, v3_norm) / (np.linalg.norm(v1_norm) * np.linalg.norm(v3_norm))
dot_prod = np.dot(v1_norm, v3_norm)
eucl_dist = np.linalg.norm(v1_norm - v3_norm)

print(f"코사인 유사도:   {cos_sim:.6f}")
print(f"내적 (= 코사인): {dot_prod:.6f}")
print(f"유클리드 거리:   {eucl_dist:.6f}")
print(f"d² = 2(1-cos):  {eucl_dist**2:.6f} ≈ {2*(1-cos_sim):.6f}")
```

```output
코사인 유사도:   0.365648
내적 (= 코사인): 0.365648
유클리드 거리:   1.126017
d² = 2(1-cos):  1.267914 ≈ 1.268704
```

정규화된 벡터에서 코사인 유사도와 내적이 정확히 같아졌고, 유클리드 거리도 $d^2 \approx 2(1-\text{cos\_sim})$ 관계를 만족합니다.

> 🔥 **실무 팁**: 정규화된 벡터에서는 내적(dot product) 계산이 가장 빠릅니다. 나눗셈이 필요 없거든요. 대규모 벡터 검색에서 성능이 중요하다면, 벡터를 미리 정규화해두고 내적으로 검색하는 것이 최적의 전략입니다.

### 개념 5: 어떤 메트릭을 선택해야 할까?

> 💡 **비유**: 옷을 고를 때 "핏(방향)"만 보는 사람이 있고, "사이즈(크기)"도 함께 보는 사람이 있죠. 코사인 유사도는 핏만 보는 것, 내적은 핏과 사이즈를 함께 보는 것, 유클리드 거리는 두 옷 사이의 전체적인 차이를 재는 것입니다.

**가장 중요한 원칙: 임베딩 모델이 학습에 사용한 메트릭을 그대로 사용하라.**

| 메트릭 | 크기 반영 | 범위 | 적합한 상황 |
|--------|----------|------|------------|
| 코사인 유사도 | ✗ (방향만) | [-1, 1] | 텍스트 의미 검색, 문서 분류, RAG |
| 유클리드 거리 | ✓ | [0, ∞) | 클러스터링, 이상 탐지, 원시 특성값 비교 |
| 내적 | ✓ | (-∞, ∞) | 추천 시스템, 정규화된 벡터의 빠른 검색 |

주요 모델별 권장 메트릭:

| 임베딩 모델 | 정규화 여부 | 권장 메트릭 |
|------------|-----------|-----------|
| OpenAI text-embedding-3-* | ✓ (L2 정규화) | 코사인 유사도 (= 내적) |
| all-MiniLM-L6-v2 | 선택 가능 | 코사인 유사도 |
| all-mpnet-base-v2 | 선택 가능 | 코사인 유사도 |
| ko-sroberta-multitask | 선택 가능 | 코사인 유사도 |

앞서 [세션 5.3](session-5.3.md)에서 다룬 Sentence Transformers 모델들은 `encode(normalize_embeddings=True)` 옵션으로 정규화할 수 있었죠. 정규화하면 내적만으로도 코사인 유사도와 동일한 결과를 얻을 수 있어 계산 비용을 줄일 수 있습니다.

```python
    "임베딩 모델은 텍스트를 벡터 공간에 매핑하여 의미를 수치로 표현합니다",
]

query = "벡터 검색에서 유사도를 측정하는 방법은?"

# 3. 임베딩 생성 (정규화 있는 버전과 없는 버전)
doc_embeddings = model.encode(documents)             # 정규화 안 함
doc_embeddings_norm = model.encode(                  # 정규화 적용
    documents, normalize_embeddings=True
)
query_embedding = model.encode(query)
query_embedding_norm = model.encode(query, normalize_embeddings=True)


# === 유사도 함수 정의 ===

def cosine_similarity_batch(query: np.ndarray, docs: np.ndarray) -> np.ndarray:
    """쿼리와 문서 배열의 코사인 유사도를 배치로 계산합니다."""
    # 각 벡터의 노름 계산
    query_norm = np.linalg.norm(query)
    doc_norms = np.linalg.norm(docs, axis=1)
    # 내적 / (쿼리 노름 * 문서 노름)
    return np.dot(docs, query) / (doc_norms * query_norm)

def euclidean_distance_batch(query: np.ndarray, docs: np.ndarray) -> np.ndarray:
    """쿼리와 문서 배열의 유클리드 거리를 배치로 계산합니다."""
    return np.linalg.norm(docs - query, axis=1)

def dot_product_batch(query: np.ndarray, docs: np.ndarray) -> np.ndarray:
    """쿼리와 문서 배열의 내적을 배치로 계산합니다."""
    return np.dot(docs, query)


# === 비정규화 벡터로 세 메트릭 비교 ===
print("=" * 65)
print("📊 비정규화 벡터에서의 세 메트릭 비교")
print("=" * 65)

# 코사인 유사도 (높을수록 유사)
cos_scores = cosine_similarity_batch(query_embedding, doc_embeddings)
cos_ranking = np.argsort(-cos_scores)  # 내림차순 정렬

# 유클리드 거리 (낮을수록 유사)
euc_distances = euclidean_distance_batch(query_embedding, doc_embeddings)
euc_ranking = np.argsort(euc_distances)  # 오름차순 정렬

# 내적 (높을수록 유사)
dot_scores = dot_product_batch(query_embedding, doc_embeddings)
dot_ranking = np.argsort(-dot_scores)  # 내림차순 정렬

print(f"\n쿼리: \"{query}\"\n")

print("순위 | 코사인 유사도      | 유클리드 거리      | 내적")
print("-" * 65)
for rank in range(len(documents)):
    ci, ei, di = cos_ranking[rank], euc_ranking[rank], dot_ranking[rank]
    print(f" {rank+1}   | [{ci}] {cos_scores[ci]:.4f}      "
          f"| [{ei}] {euc_distances[ei]:.4f}      "
          f"| [{di}] {dot_scores[di]:.4f}")


# === 정규화된 벡터로 비교 ===
print("\n" + "=" * 65)
print("📊 정규화된 벡터에서의 세 메트릭 비교")
print("=" * 65)

cos_scores_n = cosine_similarity_batch(query_embedding_norm, doc_embeddings_norm)
euc_distances_n = euclidean_distance_batch(query_embedding_norm, doc_embeddings_norm)
dot_scores_n = dot_product_batch(query_embedding_norm, doc_embeddings_norm)

cos_ranking_n = np.argsort(-cos_scores_n)
euc_ranking_n = np.argsort(euc_distances_n)
dot_ranking_n = np.argsort(-dot_scores_n)

print(f"\n쿼리: \"{query}\"\n")
print("순위 | 코사인 유사도      | 유클리드 거리      | 내적")
print("-" * 65)
for rank in range(len(documents)):
    ci, ei, di = cos_ranking_n[rank], euc_ranking_n[rank], dot_ranking_n[rank]
    print(f" {rank+1}   | [{ci}] {cos_scores_n[ci]:.4f}      "
          f"| [{ei}] {euc_distances_n[ei]:.4f}      "
          f"| [{di}] {dot_scores_n[di]:.4f}")


# === 성능 비교 ===
print("\n" + "=" * 65)
print("⚡ 성능 비교 (10,000회 반복)")
print("=" * 65)

iterations = 10000

start = time.perf_counter()
for _ in range(iterations):
    cosine_similarity_batch(query_embedding_norm, doc_embeddings_norm)
cos_time = time.perf_counter() - start

start = time.perf_counter()
for _ in range(iterations):
    euclidean_distance_batch(query_embedding_norm, doc_embeddings_norm)
euc_time = time.perf_counter() - start

start = time.perf_counter()
for _ in range(iterations):
    dot_product_batch(query_embedding_norm, doc_embeddings_norm)
dot_time = time.perf_counter() - start

print(f"코사인 유사도: {cos_time:.4f}초")
print(f"유클리드 거리: {euc_time:.4f}초")
print(f"내적:         {dot_time:.4f}초")
print(f"\n💡 내적이 코사인 유사도 대비 약 {cos_time/dot_time:.1f}배 빠릅니다")
```

이 실습의 핵심 관찰 포인트:

1. **비정규화 벡터**: 세 메트릭의 순위가 다를 수 있습니다 — 특히 내적은 벡터 크기에 영향을 받으므로 코사인 유사도와 순위가 다를 수 있습니다.
2. **정규화된 벡터**: 세 메트릭의 **순위가 모두 동일**해집니다. 코사인 유사도와 내적은 수치까지 동일하고, 유클리드 거리도 단조 변환 관계입니다.
3. **성능**: 내적이 나눗셈이 없기 때문에 가장 빠릅니다. 대규모 벡터 검색에서 이 차이가 누적되면 꽤 의미 있어요.

## 더 깊이 알아보기

### 벡터 공간 모델의 아버지: 제라드 살턴

"벡터로 유사도를 측정한다"는 아이디어는 사실 RAG나 딥러닝보다 **50년 이상** 앞선 개념입니다.

1960년대, 코넬 대학교의 **제라드 살턴(Gerard Salton)** 교수는 문서 검색을 자동화하기 위한 시스템을 개발하고 있었습니다. 당시에는 도서관 사서가 직접 키워드를 매칭해 문서를 찾아주던 시대였죠. 살턴은 혁명적인 아이디어를 냈습니다: **"문서를 벡터로 표현하고, 벡터 간의 각도로 유사도를 측정하면 어떨까?"**

1975년, 살턴은 Wong, Yang과 함께 *"A Vector Space Model for Automatic Indexing"*이라는 논문을 발표합니다. 이 논문에서 제안한 **벡터 공간 모델(Vector Space Model)**은 정보 검색(Information Retrieval)의 근간이 되었고, 그가 개발한 **SMART 시스템**은 최초의 자동 정보 검색 시스템 중 하나였습니다.

놀라운 점은, 살턴이 코사인 유사도와 TF-IDF 가중치를 결합한 방식이 이후 수십 년간 검색 엔진의 표준이 되었다는 것입니다. 우리가 지금 RAG에서 쓰는 코사인 유사도는, 1960~70년대에 도서관 문서 검색을 위해 탄생한 기법의 직계 후손인 셈이에요. 살턴은 "정보 검색의 아버지"라는 별명을 얻었고, ACM SIGIR(정보 검색 학회)에서는 그의 이름을 딴 상(Salton Award)을 지금도 수여하고 있습니다.

### 정규화의 수학: 왜 코사인 = 내적이 되는가

정규화된 벡터에서 코사인 유사도와 내적이 같아지는 이유를 수학적으로 살펴봅시다.

단위 벡터 $\hat{\mathbf{a}}$는:

$$\hat{\mathbf{a}} = \frac{\mathbf{a}}{|\mathbf{a}|}, \quad |\hat{\mathbf{a}}| = 1$$

따라서:

$$\text{cos\_sim}(\hat{\mathbf{a}}, \hat{\mathbf{b}}) = \frac{\hat{\mathbf{a}} \cdot \hat{\mathbf{b}}}{|\hat{\mathbf{a}}| \times |\hat{\mathbf{b}}|} = \frac{\hat{\mathbf{a}} \cdot \hat{\mathbf{b}}}{1 \times 1} = \hat{\mathbf{a}} \cdot \hat{\mathbf{b}}$$

유클리드 거리와의 관계도 유도할 수 있습니다:

$$d^2 = \|\hat{\mathbf{a}} - \hat{\mathbf{b}}\|^2 = |\hat{\mathbf{a}}|^2 - 2(\hat{\mathbf{a}} \cdot \hat{\mathbf{b}}) + |\hat{\mathbf{b}}|^2 = 2 - 2\cos\theta = 2(1 - \text{cos\_sim})$$

이 관계 덕분에 정규화된 벡터에서는 **어떤 메트릭을 써도 검색 순위가 동일**합니다. 그래서 벡터 데이터베이스에서 정규화된 벡터를 다룰 때는, 가장 계산이 빠른 내적(dot product)을 사용하는 것이 효율적이에요.

## 흔한 오해와 팁

> ⚠️ **흔한 오해**: "코사인 유사도가 항상 최고의 선택이다"
>
> 코사인 유사도가 텍스트 검색에서 가장 흔히 쓰이는 건 맞지만, **항상** 최선인 건 아닙니다. 예를 들어 추천 시스템에서 사용자의 활동량(벡터 크기)이 중요한 정보를 담고 있다면, 크기를 무시하는 코사인 유사도보다 내적이 더 적합합니다. 핵심은 **임베딩 모델이 학습 시 사용한 메트릭에 맞추는 것**입니다.

> 💡 **알고 계셨나요?**: OpenAI의 text-embedding-3 모델은 결과를 L2 정규화하여 반환합니다. 따라서 코사인 유사도를 쓰든 내적을 쓰든 **검색 순위가 완전히 동일**합니다. 공식 문서에서도 "The choice of distance function typically doesn't matter much"라고 안내하는 이유가 바로 이것이에요.

> 🔥 **실무 팁**: 벡터 데이터베이스의 인덱스 메트릭은 생성 후 변경할 수 없는 경우가 많습니다 (Pinecone 등). 벡터 DB를 구축하기 **전에** 반드시 임베딩 모델의 문서를 확인하여 권장 메트릭을 파악하세요. Sentence Transformers 모델은 `model.similarity_fn_name`으로 확인할 수 있습니다.

> ⚠️ **흔한 오해**: "유클리드 거리는 고차원에서 쓸모없다"
>
> "차원의 저주(curse of dimensionality)" 때문에 고차원에서 유클리드 거리의 변별력이 떨어진다는 주장이 있습니다. 이론적으로는 맞지만, 실제 임베딩 벡터는 고차원 공간에서 균등하게 분포하지 않고 **의미 있는 저차원 매니폴드 위에 집중**되어 있기 때문에, 실용적으로는 여전히 잘 작동합니다. 특히 정규화된 벡터에서는 코사인 유사도와 순위가 같으므로 걱정할 필요가 없어요.

## 핵심 정리

| 개념 | 설명 |
|------|------|
| 코사인 유사도 | 두 벡터의 방향(각도) 유사성만 측정. 크기 무시. 범위 [-1, 1] |
| 유클리드 거리 | 벡터 간 직선 거리. 크기에 민감. 범위 [0, ∞). 작을수록 유사 |
| 내적 (Dot Product) | 방향과 크기를 모두 반영. 코사인 유사도의 분자 |
| 정규화 (Normalization) | 벡터 크기를 1로 만들면, 코사인 유사도 = 내적이 됨 |
| 메트릭 선택 원칙 | 임베딩 모델 학습 시 사용된 메트릭과 동일한 것을 사용 |
| OpenAI 임베딩 | L2 정규화 적용 → 어떤 메트릭이든 순위 동일 |
| 성능 최적화 | 정규화 후 내적 사용 = 가장 빠른 유사도 계산 |
