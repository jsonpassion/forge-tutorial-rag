# 문서 유형별 특화 청킹

> 마크다운, HTML, 소스 코드 — 문서의 구조를 이해하는 스플리터로 청킹 품질을 한 단계 높이세요.

## 개요

이 섹션에서는 문서의 고유한 구조를 활용하는 **구조 인식 청킹(structure-aware chunking)** 전략을 배웁니다. [이전 섹션 4.2: 고정 크기 청킹과 재귀적 청킹](4-2)에서 학습한 `RecursiveCharacterTextSplitter`는 범용적이지만, 마크다운의 헤더 계층이나 HTML의 섹션 태그, 프로그래밍 언어의 함수/클래스 경계 같은 **문서 고유의 구조 정보를 활용하지 못합니다**. 이번 섹션에서는 LangChain이 제공하는 문서 유형별 특화 스플리터를 실습하며, 각각의 적합한 사용 시나리오를 파악합니다.

**선수 지식**: Session 4.1에서 배운 청킹의 기본 원리(chunk_size, chunk_overlap)와 Session 4.2에서 배운 `RecursiveCharacterTextSplitter`의 계층적 분리자 개념
**학습 목표**:
- `MarkdownHeaderTextSplitter`로 마크다운 헤더 기반 분할과 메타데이터 보존 방법을 이해한다
- `HTMLHeaderTextSplitter`와 `HTMLSectionSplitter`로 HTML 문서의 섹션 구조를 활용한 청킹을 수행할 수 있다
- `RecursiveCharacterTextSplitter.from_language()`로 Python, JavaScript 등 소스 코드를 함수/클래스 단위로 분할할 수 있다
- 문서 유형에 따라 적절한 스플리터를 선택하고 조합하는 판단력을 기른다

## 왜 알아야 할까?

실제 RAG 시스템에 들어오는 문서는 단순한 평문(plain text)만이 아닙니다. 기술 문서는 마크다운으로 작성되고, 웹 페이지는 HTML로 구조화되며, 코드 저장소의 소스 파일은 함수와 클래스로 조직화되어 있죠. 이런 문서를 단순히 글자 수나 줄 바꿈으로 자르면 어떻게 될까요?

마크다운 문서에서 `## API 사용법` 아래의 설명과 코드 예제가 서로 다른 청크로 쪼개지거나, Python 파일에서 함수 정의의 절반만 잘려나가는 상황이 벌어집니다. 이렇게 되면 임베딩의 의미가 왜곡되고, 검색 시 불완전한 정보만 반환되어 LLM의 답변 품질이 크게 떨어지게 됩니다.

**구조 인식 청킹**은 문서가 이미 갖고 있는 구조(헤더, 태그, 문법)를 존중하면서 분할합니다. 마치 책을 찢어서 나누는 것이 아니라, **목차를 보고 장(chapter)과 절(section) 단위로 깔끔하게 나누는 것**과 같습니다. 이 방식은 각 청크의 의미적 완결성을 보장하고, 헤더나 섹션 정보를 메타데이터로 보존하여 검색 시 필터링에도 활용할 수 있게 해줍니다.

## 핵심 개념

### 개념 1: MarkdownHeaderTextSplitter — 마크다운 헤더로 똑똑하게 나누기

> 💡 **비유**: 마크다운 문서를 나누는 것은 두꺼운 교과서를 정리하는 것과 비슷합니다. 단순히 50페이지씩 자르는 대신, **목차를 보고 "1장 서론", "2장 이론", "2.1 배경" 같은 단위로 나누는 것**이죠. 각 부분에는 어느 장, 어느 절에서 왔는지 포스트잇을 붙여둡니다.

`MarkdownHeaderTextSplitter`는 마크다운 문서의 헤더 계층(`#`, `##`, `###` 등)을 인식하여 분할합니다. 핵심은 단순 분할이 아니라, **각 청크에 해당하는 헤더 정보를 메타데이터로 자동 부착**한다는 점입니다.

```python
from langchain_text_splitters import MarkdownHeaderTextSplitter

# 분할 기준이 될 헤더 레벨과 메타데이터 키 정의
headers_to_split_on = [
    ("#", "Header 1"),    # h1 → "Header 1" 메타데이터 키
    ("##", "Header 2"),   # h2 → "Header 2" 메타데이터 키
    ("###", "Header 3"),  # h3 → "Header 3" 메타데이터 키
]

# 스플리터 생성
markdown_splitter = MarkdownHeaderTextSplitter(
    headers_to_split_on=headers_to_split_on,
    strip_headers=True,  # 기본값: 청크 본문에서 헤더 텍스트 제거
)
```

**주요 파라미터**:

| 파라미터 | 기본값 | 설명 |
|----------|--------|------|
| `headers_to_split_on` | (필수) | `(마크다운 헤더 기호, 메타데이터 키)` 튜플 리스트 |
| `strip_headers` | `True` | 청크 본문에서 헤더 라인을 제거할지 여부 |
| `return_each_line` | `False` | 각 줄을 개별 Document로 반환할지 여부 |

실제로 어떻게 동작하는지 간단한 예제로 확인해보겠습니다.

```run:python
from langchain_text_splitters import MarkdownHeaderTextSplitter

# 샘플 마크다운 문서
markdown_document = """# RAG 시스템 가이드

RAG의 기본 개념을 설명합니다.

## 검색 단계

문서를 벡터 DB에서 검색합니다.

### 임베딩 생성

텍스트를 벡터로 변환하는 과정입니다.
OpenAI의 text-embedding-3-small 모델을 주로 사용합니다.

### 유사도 검색

코사인 유사도로 가장 관련성 높은 문서를 찾습니다.

## 생성 단계

검색된 문서를 컨텍스트로 LLM에 전달합니다.
"""

headers_to_split_on = [
    ("#", "Header 1"),
    ("##", "Header 2"),
    ("###", "Header 3"),
]

splitter = MarkdownHeaderTextSplitter(headers_to_split_on=headers_to_split_on)
splits = splitter.split_text(markdown_document)

for i, doc in enumerate(splits):
    print(f"--- 청크 {i+1} ---")
    print(f"메타데이터: {doc.metadata}")
    print(f"내용: {doc.page_content[:60]}...")
    print()
```

```output
--- 청크 1 ---
메타데이터: {'Header 1': 'RAG 시스템 가이드'}
내용: RAG의 기본 개념을 설명합니다....

--- 청크 2 ---
메타데이터: {'Header 1': 'RAG 시스템 가이드', 'Header 2': '검색 단계'}
내용: 문서를 벡터 DB에서 검색합니다....

--- 청크 3 ---
메타데이터: {'Header 1': 'RAG 시스템 가이드', 'Header 2': '검색 단계', 'Header 3': '임베딩 생성'}
내용: 텍스트를 벡터로 변환하는 과정입니다.
OpenAI의 text-embedding-3-smal...

--- 청크 4 ---
메타데이터: {'Header 1': 'RAG 시스템 가이드', 'Header 2': '검색 단계', 'Header 3': '유사도 검색'}
내용: 코사인 유사도로 가장 관련성 높은 문서를 찾습니다....

--- 청크 5 ---
메타데이터: {'Header 1': 'RAG 시스템 가이드', 'Header 2': '생성 단계'}
내용: 검색된 문서를 컨텍스트로 LLM에 전달합니다....
```

결과를 보면, 각 청크에 **헤더 계층 정보가 메타데이터로 자동 부착**되는 것을 확인할 수 있습니다. "임베딩 생성" 청크에는 `Header 1`, `Header 2`, `Header 3`가 모두 포함되어 있어, 나중에 벡터 DB에서 메타데이터 필터링으로 특정 섹션의 내용만 검색하는 것이 가능해집니다.

**RecursiveCharacterTextSplitter와 조합하기**: 마크다운 헤더로 분할한 청크가 여전히 너무 크다면, 2단계로 추가 분할할 수 있습니다.

```python
from langchain_text_splitters import RecursiveCharacterTextSplitter

# 1단계: 헤더 기반 분할
md_splits = markdown_splitter.split_text(markdown_document)

# 2단계: 크기 기반 추가 분할 (메타데이터 보존!)
text_splitter = RecursiveCharacterTextSplitter(
    chunk_size=250,
    chunk_overlap=30,
)
# split_documents()를 사용해야 메타데이터가 보존됨
final_splits = text_splitter.split_documents(md_splits)
```

> ⚠️ **흔한 오해**: 2단계 분할 시 `split_text()`가 아니라 반드시 `split_documents()`를 사용해야 합니다. `split_text()`는 문자열만 받으므로 1단계에서 부착한 헤더 메타데이터가 사라집니다.

### 개념 2: HTMLHeaderTextSplitter — HTML 헤더 태그로 웹 문서 분할

> 💡 **비유**: HTML 문서를 분할하는 것은 **신문을 정리하는 것**과 같습니다. 신문에는 대제목, 소제목, 부제목이 있고, 각 기사는 이 제목들 아래에 내용이 있죠. HTML의 `<h1>`, `<h2>`, `<h3>` 태그가 바로 이 제목 역할을 합니다. 구조를 무시하고 자르면 기사 중간에서 잘리지만, 제목 단위로 나누면 각 기사가 온전히 보존됩니다.

`HTMLHeaderTextSplitter`는 HTML 문서의 헤더 태그(`<h1>` ~ `<h6>`)를 기준으로 분할하며, 마크다운 스플리터와 동일하게 헤더 정보를 메타데이터로 보존합니다.

```python
from langchain_text_splitters import HTMLHeaderTextSplitter

# HTML 헤더 태그와 메타데이터 키 매핑
headers_to_split_on = [
    ("h1", "Header 1"),
    ("h2", "Header 2"),
    ("h3", "Header 3"),
]

html_splitter = HTMLHeaderTextSplitter(headers_to_split_on=headers_to_split_on)

# HTML 문자열 또는 URL에서 직접 분할 가능
html_splits = html_splitter.split_text(html_string)
# 또는: html_splits = html_splitter.split_text_from_url("https://example.com/docs")
```

실제 HTML 문서를 분할해보겠습니다.

```run:python
from langchain_text_splitters import HTMLHeaderTextSplitter

html_string = """
<html>
<body>
    <h1>LangChain 프레임워크</h1>

    print()
```

HTML도 마크다운과 동일한 패턴으로 동작합니다. 특히 `split_text_from_url()` 메서드를 사용하면 **웹 페이지 URL을 직접 입력**하여 크롤링과 동시에 구조 인식 분할을 수행할 수 있어, 웹 문서 기반 RAG 시스템 구축 시 매우 편리합니다.

**HTMLSectionSplitter와의 차이**: LangChain은 `HTMLSectionSplitter`라는 또 다른 HTML 스플리터도 제공합니다. `HTMLHeaderTextSplitter`가 헤더 태그만을 기준으로 분할하는 반면, `HTMLSectionSplitter`는 내부적으로 XSLT 변환을 사용하여 폰트 크기까지 고려한 더 정교한 섹션 감지를 수행합니다. 단, `lxml` 패키지가 추가로 필요합니다.

```python
from langchain_text_splitters import HTMLSectionSplitter

# HTMLSectionSplitter는 폰트 크기도 고려하여 섹션 감지
headers_to_split_on = [("h1", "Header 1"), ("h2", "Header 2")]
section_splitter = HTMLSectionSplitter(headers_to_split_on=headers_to_split_on)
section_splits = section_splitter.split_text(html_string)
```

### 개념 3: 코드 스플리터 — 함수와 클래스 경계를 존중하는 분할

> 💡 **비유**: 소스 코드를 나누는 것은 **레고 설명서를 분리하는 것**과 같습니다. 레고 설명서를 한 페이지씩 자르면 하나의 조립 단계가 두 페이지에 걸쳐 잘릴 수 있습니다. 하지만 "단계 1", "단계 2" 같은 **조립 단계 단위로 나누면** 각 부분만으로도 무엇을 만드는지 이해할 수 있죠. 코드에서는 함수(`def`)와 클래스(`class`)가 바로 이 "조립 단계"입니다.

[Session 4.2](4-2)에서 배운 `RecursiveCharacterTextSplitter`는 사실 `from_language()` 클래스 메서드를 통해 **프로그래밍 언어별 전용 분리자 계층**을 자동으로 구성할 수 있습니다. 이 기능이 바로 코드 특화 청킹의 핵심입니다.

```python
from langchain_text_splitters import RecursiveCharacterTextSplitter, Language

# Python 코드용 스플리터 생성
python_splitter = RecursiveCharacterTextSplitter.from_language(
    language=Language.PYTHON,
    chunk_size=200,
    chunk_overlap=20,
)
```

`from_language()`가 내부적으로 설정하는 분리자 계층을 확인해보면, 왜 코드의 구조가 보존되는지 이해할 수 있습니다.

```run:python
from langchain_text_splitters import RecursiveCharacterTextSplitter, Language

# Python 언어의 분리자 계층 확인
python_separators = RecursiveCharacterTextSplitter.get_separators_for_language(
    Language.PYTHON
)
print("Python 분리자 계층:")
for i, sep in enumerate(python_separators):
    print(f"  {i+1}. {repr(sep)}")

print()

# JavaScript 언어의 분리자 계층 확인
js_separators = RecursiveCharacterTextSplitter.get_separators_for_language(
    Language.JS
)
print("JavaScript 분리자 계층:")
for i, sep in enumerate(js_separators):
    print(f"  {i+1}. {repr(sep)}")
```

```output
Python 분리자 계층:
  1. '\nclass '
  2. '\ndef '
  3. '\n\tdef '
  4. '\n\n'
  5. '\n'
  6. ' '
  7. ''

JavaScript 분리자 계층:
  1. '\nfunction '
  2. '\nconst '
  3. '\nlet '
  4. '\nvar '
  5. '\nclass '
  6. '\nif '
  7. '\nfor '
  8. '\nwhile '
  9. '\nswitch '
  10. '\ncase '
  11. '\ndefault '
  12. '\n\n'
  13. '\n'
  14. ' '
  15. ''
```

Python의 경우 `\nclass ` → `\ndef ` → `\n\tdef ` 순서로 시도합니다. 즉, **클래스 단위로 나누는 것을 가장 먼저 시도**하고, 그래도 크면 함수 단위, 메서드 단위, 빈 줄, 줄 바꿈 순서로 점점 작은 단위로 분할하는 것이죠. [Session 4.2](4-2)에서 배운 재귀적 분할의 원리가 프로그래밍 언어의 문법 구조에 맞게 적용된 것입니다.

**지원하는 언어 목록**: `Language` enum은 Python, JavaScript, TypeScript, Java, Go, Rust, C/C++, C#, Ruby, Scala, Swift, Kotlin, PHP, Lua, Perl, Haskell, Elixir 등 **25개 이상의 언어**를 지원합니다. Markdown, HTML, LaTeX, reStructuredText 같은 마크업 언어도 포함됩니다.

실제 Python 코드를 분할해보겠습니다.

```run:python

            return f.read()

    \"\"\"프로세서 팩토리 함수\"\"\"
```

### 개념 4: 스플리터 선택 가이드 — 언제 어떤 스플리터를 쓸까?

실무에서는 하나의 RAG 시스템에 다양한 유형의 문서가 들어옵니다. 문서 유형을 감지하고 적절한 스플리터를 선택하는 전략이 필요합니다.

| 문서 유형 | 추천 스플리터 | 이유 |
|-----------|---------------|------|
| 마크다운 (`.md`) | `MarkdownHeaderTextSplitter` + `RecursiveCharacterTextSplitter` | 헤더 계층 보존 + 크기 제어 |
| HTML 웹 페이지 | `HTMLHeaderTextSplitter` + `RecursiveCharacterTextSplitter` | 헤더 태그 기반 분할 + 크기 제어 |
| 복잡한 HTML (폰트 크기 등) | `HTMLSectionSplitter` | XSLT 기반 정교한 섹션 감지 |
| Python 소스 코드 | `from_language(Language.PYTHON)` | 클래스/함수 경계 보존 |
| JavaScript 코드 | `from_language(Language.JS)` | 함수/변수 선언 경계 보존 |
| 일반 텍스트 | `RecursiveCharacterTextSplitter` | 범용적 재귀 분할 |
| 혼합 문서 (여러 유형) | 파일 확장자 기반 라우팅 | 유형별 최적 스플리터 자동 선택 |

> 🔥 **실무 팁**: 대부분의 프로덕션 RAG 시스템에서는 **2단계 분할** 패턴을 사용합니다. 1단계에서 구조 인식 스플리터(마크다운/HTML/코드)로 의미 단위를 나누고, 2단계에서 `RecursiveCharacterTextSplitter`로 청크 크기를 일정하게 맞추는 방식입니다. 이렇게 하면 의미적 완결성과 일관된 크기를 동시에 달성할 수 있습니다.

## 실습: 직접 해보기

이제 다양한 문서 유형이 섞인 실제 시나리오를 구현해보겠습니다. 파일 확장자에 따라 적절한 스플리터를 자동으로 선택하는 **문서 유형 인식 청킹 파이프라인**을 만들어봅시다.

```python
"""
문서 유형별 자동 청킹 파이프라인
- 마크다운, HTML, Python 코드를 각각 최적의 방식으로 분할
- 2단계 분할: 구조 인식 → 크기 제어
"""
from langchain_text_splitters import (
    MarkdownHeaderTextSplitter,
    HTMLHeaderTextSplitter,
    RecursiveCharacterTextSplitter,
    Language,
)
from langchain_core.documents import Document


# === 1. 각 유형별 스플리터 설정 ===

# 마크다운용: 헤더 기반 1차 분할
md_headers = [
    ("#", "Header 1"),
    ("##", "Header 2"),
    ("###", "Header 3"),
]
md_splitter = MarkdownHeaderTextSplitter(
    headers_to_split_on=md_headers,
    strip_headers=False,  # 헤더를 본문에 유지하여 임베딩에 포함
)

# HTML용: 헤더 태그 기반 1차 분할
html_headers = [
    ("h1", "Header 1"),
    ("h2", "Header 2"),
    ("h3", "Header 3"),
]
html_splitter = HTMLHeaderTextSplitter(headers_to_split_on=html_headers)

# Python 코드용: 함수/클래스 경계 인식 분할
python_splitter = RecursiveCharacterTextSplitter.from_language(
    language=Language.PYTHON,
    chunk_size=500,
    chunk_overlap=50,
)

# 2단계 공통: 크기 기반 추가 분할
size_splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,
    chunk_overlap=50,
)


# === 2. 문서 유형 감지 및 라우팅 ===

def chunk_document(file_path: str, content: str) -> list[Document]:
    """파일 확장자에 따라 최적의 스플리터를 선택하여 청킹합니다."""
    
    if file_path.endswith(".md"):
        # 1단계: 마크다운 헤더 기반 분할
        header_splits = md_splitter.split_text(content)
        # 2단계: 크기 제어 (메타데이터 보존을 위해 split_documents 사용)
        final_splits = size_splitter.split_documents(header_splits)
        
    elif file_path.endswith((".html", ".htm")):
        # 1단계: HTML 헤더 태그 기반 분할
        header_splits = html_splitter.split_text(content)
        # 2단계: 크기 제어
        final_splits = size_splitter.split_documents(header_splits)
        
    elif file_path.endswith(".py"):
        # 코드 스플리터는 이미 크기 제어가 내장됨 (1단계만으로 충분)
        final_splits = python_splitter.create_documents(
            [content],
            metadatas=[{"source": file_path, "type": "python"}],
        )
        
    else:
        # 범용 폴백: RecursiveCharacterTextSplitter
        final_splits = size_splitter.create_documents(
            [content],
            metadatas=[{"source": file_path, "type": "text"}],
        )
    
    # 모든 청크에 소스 파일 정보 추가
    for doc in final_splits:
        doc.metadata["source"] = file_path
    
    return final_splits


# === 3. 테스트: 다양한 문서 유형으로 실행 ===

# 마크다운 문서
md_content = """# 벡터 데이터베이스 가이드

## ChromaDB 소개

ChromaDB는 오픈소스 임베딩 데이터베이스입니다.
로컬 환경에서 빠르게 시작할 수 있어 프로토타이핑에 최적입니다.
Python 네이티브 API를 제공하며, 설치도 pip install chromadb 한 줄이면 됩니다.

### 기본 사용법

컬렉션을 생성하고 문서를 추가하는 방법은 간단합니다.
"""

# Python 코드
py_content = """
class VectorStore:
    def __init__(self, collection_name: str):
        self.collection_name = collection_name
        self.documents = []
    
    def add_documents(self, docs: list[str]) -> None:
        for doc in docs:
            self.documents.append(doc)

def search(query: str, top_k: int = 5) -> list[str]:
    return []
"""

# 마크다운 청킹 결과 확인
print("=== 마크다운 청킹 결과 ===")
md_chunks = chunk_document("guide.md", md_content)
for i, chunk in enumerate(md_chunks):
    print(f"\n청크 {i+1}: {chunk.metadata}")
    print(f"  내용: {chunk.page_content[:80]}...")

print("\n\n=== Python 코드 청킹 결과 ===")
py_chunks = chunk_document("vector_store.py", py_content)
for i, chunk in enumerate(py_chunks):
    print(f"\n청크 {i+1}: {chunk.metadata}")
    print(f"  내용: {chunk.page_content[:80]}...")

# 전체 통계
total = len(md_chunks) + len(py_chunks)
print(f"\n\n총 {total}개 청크 생성 (마크다운: {len(md_chunks)}, Python: {len(py_chunks)})")
```

이 파이프라인은 실제 프로덕션 RAG 시스템의 청킹 계층 설계를 축소한 것입니다. 파일 확장자 기반 라우팅을 MIME 타입이나 콘텐츠 감지 라이브러리(예: `python-magic`)로 대체하면 더 견고해집니다.

## 더 깊이 알아보기

### "구조를 이해하는 파서"의 역사 — HTML과 DOM의 탄생

문서의 구조를 프로그래밍적으로 다루는 아이디어는 HTML 자체의 역사와 맞닿아 있습니다. 1991년 팀 버너스-리(Tim Berners-Lee)가 CERN에서 HTML을 만들 때, 핵심 아이디어는 **문서에 구조적 의미(semantic structure)를 부여**하는 것이었습니다. `<h1>`이 "큰 글씨"가 아니라 "문서의 최상위 제목"이라는 **의미**를 갖는다는 것이죠.

이후 1998년, W3C가 DOM(Document Object Model) 표준을 발표하면서 HTML 문서를 **트리 구조로 프로그래밍적으로 탐색하고 조작**할 수 있게 되었습니다. LangChain의 `HTMLHeaderTextSplitter`가 BeautifulSoup으로 HTML을 파싱하고, `HTMLSectionSplitter`가 XSLT 변환을 사용하는 것도 이 DOM 트리의 개념을 활용한 것입니다.

마크다운은 2004년 존 그루버(John Gruber)와 아론 스워츠(Aaron Swartz)가 만들었는데, "이메일의 평문 포맷팅 규칙을 체계화하자"는 아이디어에서 출발했습니다. `#`으로 헤더를 표현하는 직관적 문법은 HTML의 `<h1>` 태그를 더 쓰기 쉽게 만든 것이죠. 흥미롭게도 마크다운의 공식 명세가 없어 방언이 난립하다가, 2014년에야 CommonMark라는 표준 명세가 등장했습니다. LangChain의 `MarkdownHeaderTextSplitter`는 이 `#` 기호 패턴을 기반으로 헤더를 감지합니다.

### 코드 스플리팅의 배경 — AST에서 정규식까지

소스 코드를 의미 단위로 나누려면 이상적으로는 **AST(Abstract Syntax Tree, 추상 구문 트리)**를 파싱해야 합니다. 컴파일러가 코드를 이해하는 방식 그대로 함수, 클래스, 블록 경계를 정확히 파악할 수 있으니까요. 하지만 AST 파싱은 언어별로 전용 파서가 필요하고, 문법 오류가 있는 코드는 파싱에 실패합니다.

LangChain의 `from_language()` 접근법은 흥미로운 실용적 타협입니다. 완벽한 AST 파싱 대신, **각 언어의 핵심 키워드 패턴을 분리자로 사용**하는 것이죠. Python의 `\nclass `나 `\ndef ` 같은 패턴은 AST만큼 정확하지는 않지만, 대부분의 실무 코드에서 충분히 잘 동작하며 **언어별 파서 없이도 25개 이상의 언어를 지원**할 수 있게 해줍니다. 완벽함보다 실용성을 택한 엔지니어링 트레이드오프의 좋은 예시입니다.

## 흔한 오해와 팁

> ⚠️ **흔한 오해**: "구조 인식 스플리터만 쓰면 `chunk_size` 제어가 자동으로 된다"고 생각하기 쉽습니다. 하지만 `MarkdownHeaderTextSplitter`와 `HTMLHeaderTextSplitter`는 **구조 단위로만 분할**할 뿐, 청크 크기를 제한하지 않습니다. 하나의 섹션이 10,000자라면 10,000자짜리 청크가 그대로 나옵니다. 반드시 `RecursiveCharacterTextSplitter`로 2단계 분할을 추가하세요.

> 💡 **알고 계셨나요?**: `Language` enum에는 프로그래밍 언어뿐 아니라 `Language.MARKDOWN`, `Language.HTML`, `Language.LATEX` 같은 마크업 언어도 포함되어 있습니다. 즉, `RecursiveCharacterTextSplitter.from_language(Language.MARKDOWN)`으로도 마크다운을 분할할 수 있는데요, 이 방식은 `MarkdownHeaderTextSplitter`와 달리 **헤더 메타데이터를 생성하지 않고** 단순히 마크다운 문법 패턴을 분리자로 사용합니다. 메타데이터가 필요하면 `MarkdownHeaderTextSplitter`, 단순 분할이면 `from_language(Language.MARKDOWN)`을 선택하세요.

> 🔥 **실무 팁**: `MarkdownHeaderTextSplitter`의 `strip_headers` 파라미터 선택은 RAG 검색 품질에 영향을 줍니다. `strip_headers=True`(기본값)로 설정하면 청크 본문이 깔끔해지지만, 임베딩에 헤더 정보가 포함되지 않아 검색 시 맥락이 부족할 수 있습니다. **`strip_headers=False`로 설정하면 헤더가 본문에 남아 임베딩에 반영**되므로, "API 사용법"이라는 헤더 아래의 내용을 "API"로 검색했을 때 더 잘 매칭됩니다. 동시에 메타데이터에도 헤더가 남아 필터링도 가능하죠. 대부분의 RAG 시나리오에서는 `strip_headers=False`를 권장합니다.

## 핵심 정리

| 개념 | 설명 |
|------|------|
| `MarkdownHeaderTextSplitter` | 마크다운 `#`, `##`, `###` 헤더 기준 분할. 헤더 정보를 메타데이터로 자동 보존 |
| `HTMLHeaderTextSplitter` | HTML `<h1>`~`<h6>` 헤더 태그 기준 분할. URL 직접 입력 지원 (`split_text_from_url`) |
| `HTMLSectionSplitter` | XSLT 기반 정교한 HTML 섹션 감지. 폰트 크기까지 고려. `lxml` 필요 |
| `from_language()` | `RecursiveCharacterTextSplitter`의 클래스 메서드. 25개+ 언어별 최적 분리자 자동 설정 |
| `Language` enum | `Language.PYTHON`, `Language.JS` 등 지원 언어 열거형. 마크업 언어도 포함 |
| 2단계 분할 패턴 | 1단계: 구조 인식 스플리터 → 2단계: `RecursiveCharacterTextSplitter`로 크기 제어 |
| `strip_headers` | `MarkdownHeaderTextSplitter`의 헤더 제거 옵션. RAG에서는 `False` 권장 |
| `split_documents()` | 2단계 분할 시 메타데이터를 보존하려면 `split_text()`가 아닌 이 메서드 사용 |

지금까지 문자 수 기반, 재귀적, 구조 인식 등 다양한 청킹 전략을 배웠습니다. 하지만 이 모든 방식에는 공통적인 한계가 있습니다 — **텍스트의 "의미"를 직접 이해하지 못한다는 것**이죠. 다음 섹션 [4.4: 시멘틱 청킹](4-4)에서는 임베딩 모델을 활용하여 **의미적 유사성이 변하는 지점을 감지하고 그 경계에서 분할**하는 시멘틱 청킹(Semantic Chunking)을 학습합니다. 규칙 기반에서 AI 기반으로, 청킹 전략의 패러다임이 전환되는 순간을 경험하게 될 것입니다.
