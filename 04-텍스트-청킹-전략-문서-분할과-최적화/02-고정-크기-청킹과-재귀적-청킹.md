# 고정 크기 청킹과 재귀적 청킹

> CharacterTextSplitter와 RecursiveCharacterTextSplitter의 동작 원리를 비교하고, 재귀적 분할 전략의 우월함을 코드로 체험합니다.

## 개요

이 섹션에서는 LangChain이 제공하는 두 가지 대표적인 텍스트 스플리터의 **내부 동작 원리**를 파헤칩니다. 단순한 고정 크기 분할이 왜 문제를 일으키는지 직접 확인하고, 재귀적 분할이 어떤 메커니즘으로 이를 해결하는지 단계별로 이해합니다.

**선수 지식**: [4.1 청킹의 중요성과 기본 원리](04-텍스트-청킹-전략-문서-분할과-최적화/01-청킹의-중요성과-기본-원리.md)에서 배운 chunking, chunk_size, chunk_overlap, semantic_coherence 개념
**학습 목표**:
- CharacterTextSplitter의 단일 분리자 방식과 그 한계를 설명할 수 있다
- RecursiveCharacterTextSplitter의 계층적 분리자 전략(`\n\n` → `\n` → ` ` → `""`)을 이해한다
- chunk_size와 chunk_overlap 파라미터 조합이 결과에 미치는 영향을 실험적으로 확인한다
- 문서 특성에 따라 적절한 스플리터를 선택할 수 있다

## 왜 알아야 할까?

앞서 [4.1 청킹의 중요성과 기본 원리](04-텍스트-청킹-전략-문서-분할과-최적화/01-청킹의-중요성과-기본-원리.md)에서 청킹이 RAG 검색 품질에 결정적이라는 것을 배웠죠. 그런데 실제로 코드를 짜다 보면 "어떤 스플리터를 써야 하지?"라는 질문에 부딪히게 됩니다.

LangChain에는 10개가 넘는 텍스트 스플리터가 있는데요, 그중에서도 `CharacterTextSplitter`와 `RecursiveCharacterTextSplitter`는 가장 기본적이면서도 가장 많이 쓰이는 두 가지입니다. 놀랍게도, 이 둘의 동작 방식은 근본적으로 다르거든요.

잘못된 스플리터 선택은 검색 recall을 최대 9%까지 떨어뜨릴 수 있다는 연구 결과도 있습니다. 단순히 "잘라서 넣으면 되겠지"라는 생각이 얼마나 위험한지, 이 세션에서 직접 눈으로 확인해보겠습니다.

## 핵심 개념

### 개념 1: CharacterTextSplitter — 칼로 자르는 방식

> 💡 **비유**: CharacterTextSplitter는 긴 식빵을 **일정한 간격으로 칼로 자르는 것**과 같습니다. 빵 사이에 건포도가 있든, 크림이 있든 상관없이 정해진 위치에서 정확히 잘라버리죠. 간단하지만, 건포도가 반으로 잘릴 수 있습니다.

`CharacterTextSplitter`는 **단일 분리자(separator)**를 기준으로 텍스트를 분할합니다. 기본 분리자는 `"\n\n"`(이중 줄바꿈)이고, 이 분리자로 텍스트를 나눈 뒤 `chunk_size`에 맞게 병합하거나 그대로 반환합니다.

핵심적인 특징은 다음과 같습니다:

- **단일 분리자만 사용**: 지정한 하나의 분리자로만 자릅니다
- **분리자가 없으면 무력**: 텍스트에 `"\n\n"`이 없으면 텍스트 전체가 하나의 청크가 됩니다
- **chunk_size 초과 가능**: 분리자 사이의 텍스트가 `chunk_size`보다 길면 그대로 반환될 수 있습니다

```python
from langchain_text_splitters import CharacterTextSplitter

# 단일 분리자(\n\n)로만 분할
splitter = CharacterTextSplitter(
    separator="\n\n",      # 이 분리자 하나로만 자름
    chunk_size=200,         # 목표 청크 크기 (문자 수)
    chunk_overlap=20,       # 청크 간 겹침
    length_function=len,    # 길이 측정 함수
)
```

여기서 중요한 점이 있습니다. `CharacterTextSplitter`는 분리자로 텍스트를 나눈 뒤, 나눠진 조각들을 `chunk_size` 이하가 되도록 **합치는(merge)** 작업을 합니다. 하지만 분리자 사이의 한 조각이 이미 `chunk_size`보다 크다면? 그 조각은 더 이상 쪼개지 않고 그대로 반환됩니다.

### 개념 2: RecursiveCharacterTextSplitter — 단계별로 내려가는 방식

> 💡 **비유**: RecursiveCharacterTextSplitter는 **책을 정리하는 사서**와 같습니다. 먼저 **챕터(단락)** 단위로 나눠보고, 챕터가 너무 길면 **절(문장)** 단위로 다시 나누고, 그래도 길면 **단어** 단위로 나눕니다. 항상 가장 의미 있는 큰 단위부터 시도하죠.

`RecursiveCharacterTextSplitter`는 LangChain의 **권장 텍스트 스플리터**입니다. 이름에 "Recursive(재귀적)"가 들어간 이유가 있는데요, 분리자 목록을 **계층적으로 순회**하면서 재귀적으로 분할하기 때문입니다.

기본 분리자 계층은 다음과 같습니다:

| 우선순위 | 분리자 | 의미 | 분할 단위 |
|:---:|:---:|---|---|
| 1 | `"\n\n"` | 이중 줄바꿈 | 단락(paragraph) |
| 2 | `"\n"` | 줄바꿈 | 줄(line) |
| 3 | `" "` | 공백 | 단어(word) |
| 4 | `""` | 빈 문자열 | 문자(character) |

알고리즘의 동작 과정을 단계별로 살펴보겠습니다:

1. **1차 시도**: `"\n\n"`으로 텍스트를 나눕니다 (단락 분리)
2. **크기 확인**: 각 조각이 `chunk_size` 이하인지 확인합니다
3. **재귀 호출**: `chunk_size`를 초과하는 조각이 있으면, 그 조각에 대해 다음 분리자(`"\n"`)로 다시 분할합니다
4. **반복**: 여전히 큰 조각이 있으면 `" "`로, 그래도 크면 `""`로 한 글자씩 나눕니다
5. **병합**: 작아진 조각들을 `chunk_size`에 맞게 다시 합칩니다

```python
from langchain_text_splitters import RecursiveCharacterTextSplitter

# 계층적 분리자로 재귀 분할
splitter = RecursiveCharacterTextSplitter(
    separators=["\n\n", "\n", " ", ""],  # 분리자 계층 (기본값)
    chunk_size=200,          # 목표 청크 크기
    chunk_overlap=20,        # 청크 간 겹침
    length_function=len,     # 길이 측정 함수
    is_separator_regex=False # 분리자가 정규식인지 여부
)
```

### 개념 3: 두 스플리터의 결정적 차이 — 직접 비교

두 스플리터의 차이를 확실하게 이해하기 위해, 같은 텍스트에 두 스플리터를 적용해보겠습니다.

```run:python
from langchain_text_splitters import (
    CharacterTextSplitter,
    RecursiveCharacterTextSplitter
)

# 테스트용 텍스트 (단락, 줄바꿈, 문장이 혼합된 구조)
text = """RAG는 검색 증강 생성의 약자입니다. LLM의 한계를 극복하기 위해 등장했습니다.

RAG의 핵심 구성 요소는 세 가지입니다.
첫째, 문서를 청크로 분할합니다.
둘째, 임베딩으로 벡터화합니다.
셋째, 유사도 검색으로 관련 문서를 찾습니다.

이렇게 검색된 문서를 LLM의 프롬프트에 포함시켜 더 정확한 답변을 생성합니다. 이것이 RAG의 기본 원리입니다."""

# CharacterTextSplitter (단일 분리자: \n\n)
char_splitter = CharacterTextSplitter(
    separator="\n\n",
    chunk_size=100,
    chunk_overlap=0,
)
char_chunks = char_splitter.split_text(text)

print("=== CharacterTextSplitter ===")
for i, chunk in enumerate(char_chunks):
    print(f"\n[청크 {i+1}] (길이: {len(chunk)}자)")
    print(chunk)
```

```output

[청크 1] (길이: 47자)
RAG는 검색 증강 생성의 약자입니다. LLM의 한계를 극복하기 위해 등장했습니다.

[청크 2] (길이: 100자)
RAG의 핵심 구성 요소는 세 가지입니다.
첫째, 문서를 청크로 분할합니다.
둘째, 임베딩으로 벡터화합니다.
셋째, 유사도 검색으로 관련 문서를 찾습니다.

[청크 3] (길이: 64자)
이렇게 검색된 문서를 LLM의 프롬프트에 포함시켜 더 정확한 답변을 생성합니다. 이것이 RAG의 기본 원리입니다.
```

주목하세요. `CharacterTextSplitter`는 `"\n\n"`으로만 나누기 때문에, 청크 2의 길이가 100자인데도 내부의 줄바꿈(`\n`)은 무시합니다. 만약 이 단락이 500자였다면? `chunk_size=100`인데도 500자짜리 청크가 그대로 나옵니다.

이제 같은 텍스트에 `RecursiveCharacterTextSplitter`를 적용해보겠습니다:

```run:python
from langchain_text_splitters import RecursiveCharacterTextSplitter

text = """RAG는 검색 증강 생성의 약자입니다. LLM의 한계를 극복하기 위해 등장했습니다.

RAG의 핵심 구성 요소는 세 가지입니다.
첫째, 문서를 청크로 분할합니다.

이렇게 검색된 문서를 LLM의 프롬프트에 포함시켜 더 정확한 답변을 생성합니다. 이것이 RAG의 기본 원리입니다."""

# RecursiveCharacterTextSplitter (계층적 분리자)
recursive_splitter = RecursiveCharacterTextSplitter(
    chunk_size=100,
    chunk_overlap=0,
)
recursive_chunks = recursive_splitter.split_text(text)

print("=== RecursiveCharacterTextSplitter ===")
for i, chunk in enumerate(recursive_chunks):
    print(f"\n[청크 {i+1}] (길이: {len(chunk)}자)")
    print(chunk)
```

```output

[청크 2] (길이: 72자)

```

차이가 보이시나요? `RecursiveCharacterTextSplitter`는 두 번째 단락이 `chunk_size(100)`를 넘자, 자동으로 `"\n"`(줄바꿈)을 기준으로 한 번 더 분할했습니다. 덕분에 모든 청크가 100자 이하로 유지되면서도 문장 단위의 의미를 보존합니다.

### 개념 4: chunk_overlap — 맥락의 다리 놓기

> 💡 **비유**: chunk_overlap은 **퍼즐 조각의 겹치는 부분**과 같습니다. 퍼즐 조각이 서로 조금씩 겹쳐야 전체 그림을 이어 붙일 수 있듯이, 청크도 일부가 겹쳐야 잘린 맥락을 복원할 수 있습니다.

`chunk_overlap`은 인접한 청크 사이에 공유되는 텍스트의 양을 지정합니다. 이 파라미터는 청킹 과정에서 잘려나가는 맥락을 보전하는 핵심 장치입니다.

```run:python
from langchain_text_splitters import RecursiveCharacterTextSplitter

text = "벡터 데이터베이스는 고차원 벡터를 저장합니다. 유사도 검색을 빠르게 수행하는 것이 핵심입니다. HNSW 알고리즘이 대표적인 인덱싱 방법이며, 대부분의 벡터 DB에서 기본으로 사용됩니다."

# overlap 없이 분할
splitter_no_overlap = RecursiveCharacterTextSplitter(
    chunk_size=80, chunk_overlap=0
)
# overlap 있게 분할
splitter_with_overlap = RecursiveCharacterTextSplitter(
    chunk_size=80, chunk_overlap=20
)

print("=== overlap=0 ===")
for i, chunk in enumerate(splitter_no_overlap.split_text(text)):
    print(f"[{i+1}] {chunk}")

print("\n=== overlap=20 ===")
for i, chunk in enumerate(splitter_with_overlap.split_text(text)):
    print(f"[{i+1}] {chunk}")
```

```output
=== overlap=0 ===
[1] 벡터 데이터베이스는 고차원 벡터를 저장합니다. 유사도 검색을 빠르게 수행하는 것이 핵심입니다.
[2] HNSW 알고리즘이 대표적인 인덱싱 방법이며, 대부분의 벡터 DB에서 기본으로 사용됩니다.

=== overlap=20 ===
[1] 벡터 데이터베이스는 고차원 벡터를 저장합니다. 유사도 검색을 빠르게 수행하는 것이 핵심입니다.
[2] 것이 핵심입니다. HNSW 알고리즘이 대표적인 인덱싱 방법이며, 대부분의 벡터 DB에서 기본으로 사용됩니다.
```

`overlap=20`인 경우, 청크 2의 시작 부분에 "것이 핵심입니다."가 포함되어 청크 1과의 맥락이 자연스럽게 연결됩니다. 실무에서는 **chunk_size의 10~20%**를 overlap으로 설정하는 것이 일반적입니다. NVIDIA의 2024년 벤치마크 연구에서는 **15% overlap**이 최적의 성능을 보였습니다.

### 개념 5: 한국어 텍스트를 위한 분리자 커스터마이징

기본 분리자 `["\n\n", "\n", " ", ""]`는 영어에 최적화되어 있습니다. 한국어 텍스트에서는 마침표(`.`)나 한국어 구두점(`。`)을 분리자에 추가하면 더 자연스러운 분할이 가능합니다.

```python
from langchain_text_splitters import RecursiveCharacterTextSplitter

# 한국어에 최적화된 분리자 계층
korean_splitter = RecursiveCharacterTextSplitter(
    separators=[
        "\n\n",   # 단락 구분
        "\n",     # 줄바꿈
        ". ",     # 마침표 + 공백 (문장 경계)
        "다. ",   # 한국어 종결어미 패턴
        " ",      # 공백
        "",       # 문자 단위 (최후의 수단)
    ],
    chunk_size=300,
    chunk_overlap=50,
)
```

> ⚠️ **흔한 오해**: 분리자 순서는 상관없다? 아닙니다! 분리자 리스트에서 **앞쪽에 있는 분리자가 먼저 시도**됩니다. 큰 단위(단락)부터 작은 단위(문자)로 배치해야 의미적 완결성이 보존됩니다.

## 실습: 직접 해보기

실제 문서 텍스트를 두 스플리터로 분할하고, 결과를 비교 분석하는 완전한 실습입니다.

```run:python
from langchain_text_splitters import (
    CharacterTextSplitter,
    RecursiveCharacterTextSplitter
)

# 실제 기술 문서 스타일의 텍스트
document = """# RAG 시스템의 핵심 컴포넌트

## 1. 문서 로더 (Document Loader)

RAG 시스템의 첫 번째 단계는 다양한 소스에서 문서를 로딩하는 것입니다. PDF, 웹페이지, 데이터베이스 등 다양한 형식의 데이터를 통일된 Document 객체로 변환합니다.

LangChain은 100개 이상의 문서 로더를 제공합니다. PyPDFLoader, WebBaseLoader, CSVLoader 등이 대표적이며, 각 로더는 특정 데이터 소스에 최적화되어 있습니다.

## 2. 텍스트 스플리터 (Text Splitter)

로딩된 문서는 보통 수천~수만 자에 달합니다. 이를 임베딩 모델의 토큰 제한에 맞게 잘라야 하는데, 단순히 자르면 문맥이 깨집니다. 텍스트 스플리터는 의미적 경계를 존중하면서 문서를 분할합니다.

RecursiveCharacterTextSplitter가 가장 범용적이며, 문서 구조를 파악하여 단락, 문장, 단어 순서로 계층적으로 분할합니다.

## 3. 임베딩 모델 (Embedding Model)

분할된 청크를 고차원 벡터로 변환합니다. OpenAI의 text-embedding-3-small이나 오픈소스 모델인 sentence-transformers를 많이 사용합니다.

임베딩의 핵심은 의미적으로 유사한 텍스트를 벡터 공간에서 가까운 위치에 배치하는 것입니다. 이를 통해 사용자 질문과 관련된 청크를 빠르게 찾을 수 있습니다."""

# === 실험 1: 두 스플리터 비교 ===
print("=" * 60)
print("실험 1: CharacterTextSplitter vs Recursive (chunk_size=200)")
print("=" * 60)

char_splitter = CharacterTextSplitter(
    separator="\n\n",
    chunk_size=200,
    chunk_overlap=0,
)
recursive_splitter = RecursiveCharacterTextSplitter(
    chunk_size=200,
    chunk_overlap=0,
)

char_chunks = char_splitter.split_text(document)
recursive_chunks = recursive_splitter.split_text(document)

print(f"\nCharacterTextSplitter: {len(char_chunks)}개 청크")
for i, chunk in enumerate(char_chunks):
    print(f"  청크 {i+1}: {len(chunk)}자")

print(f"\nRecursiveCharacterTextSplitter: {len(recursive_chunks)}개 청크")
for i, chunk in enumerate(recursive_chunks):
    print(f"  청크 {i+1}: {len(chunk)}자")

# === 실험 2: chunk_size 변화에 따른 결과 ===
print("\n" + "=" * 60)
print("실험 2: chunk_size별 청크 수 비교 (RecursiveCharacterTextSplitter)")
print("=" * 60)

for size in [100, 200, 300, 500]:
    splitter = RecursiveCharacterTextSplitter(
        chunk_size=size, chunk_overlap=0
    )
    chunks = splitter.split_text(document)
    avg_len = sum(len(c) for c in chunks) / len(chunks)
    print(f"  chunk_size={size:>3} → {len(chunks):>2}개 청크, 평균 {avg_len:.0f}자")

# === 실험 3: chunk_overlap 효과 ===
print("\n" + "=" * 60)
print("실험 3: overlap 비율별 청크 수 (chunk_size=200)")
print("=" * 60)

for overlap in [0, 20, 40, 60]:
    splitter = RecursiveCharacterTextSplitter(
        chunk_size=200, chunk_overlap=overlap
    )
    chunks = splitter.split_text(document)
    ratio = overlap / 200 * 100
    print(f"  overlap={overlap:>2} ({ratio:>4.0f}%) → {len(chunks):>2}개 청크")
```

## 더 깊이 알아보기

### 텍스트 청킹의 "5단계 레벨" — Greg Kamradt의 프레임워크

텍스트 청킹 전략을 체계적으로 정리한 것으로 유명한 사람이 바로 **Greg Kamradt**입니다. 2023년에 그가 공개한 "5 Levels of Text Splitting" 프레임워크는 RAG 커뮤니티에 큰 영향을 미쳤는데요, 이 프레임워크는 청킹 전략을 다섯 단계로 분류합니다:

| 레벨 | 전략 | 설명 |
|:---:|---|---|
| 1 | Character Split | 고정 문자 수로 분할 (가장 단순) |
| 2 | Recursive Character | 계층적 분리자로 재귀 분할 |
| 3 | Document-Specific | 마크다운, 코드 등 문서 형식별 분할 |
| 4 | Semantic Chunking | 임베딩 유사도 기반 의미 단위 분할 |
| 5 | Agentic Chunking | AI 에이전트가 판단하는 분할 |

오늘 우리가 배운 `CharacterTextSplitter`가 레벨 1, `RecursiveCharacterTextSplitter`가 레벨 2에 해당합니다. 이 프레임워크에서 Greg Kamradt가 강조한 핵심 철학이 있는데요: **"청킹의 목표는 자르는 것 자체가 아니라, 나중에 가치 있게 검색될 수 있는 형태로 데이터를 준비하는 것"**이라는 점입니다.

LangChain의 `SemanticChunker`(레벨 4)도 사실 Greg Kamradt의 노트북에서 영감을 받아 구현되었습니다. 이 주제는 [4.4 시멘틱 청킹](04-텍스트-청킹-전략-문서-분할과-최적화/04-시멘틱-청킹-의미-기반-분할.md)에서 자세히 다룹니다.

### 왜 "재귀적"이라는 이름이 붙었을까?

컴퓨터 과학에서 **재귀(recursion)**란 함수가 자기 자신을 호출하는 것을 말합니다. `RecursiveCharacterTextSplitter`의 내부 구현을 보면, `_split_text()` 메서드가 분리자 리스트의 다음 원소를 가지고 자기 자신을 다시 호출합니다. 마치 러시아 인형(마트료시카)처럼, 큰 분할 안에서 더 작은 분할이 이루어지는 구조인 거죠.

이 설계가 영리한 이유는, 어떤 텍스트가 들어와도 결국 `chunk_size` 이하의 조각으로 만들 수 있다는 **보장**이 있기 때문입니다. 최후의 분리자 `""`가 한 글자 단위로 쪼개기 때문이죠.

## 흔한 오해와 팁

> ⚠️ **흔한 오해**: "CharacterTextSplitter의 `chunk_size=500`이면 모든 청크가 500자 이하로 나온다." 아닙니다! `CharacterTextSplitter`는 분리자(`\n\n`) 사이의 텍스트가 500자를 넘어도 그 이상 쪼개지 않습니다. 반면 `RecursiveCharacterTextSplitter`는 다음 레벨의 분리자로 재귀 분할하여 `chunk_size` 제약을 최대한 지킵니다.

> 💡 **알고 계셨나요?**: LangChain에는 코드 전용 스플리터도 있습니다. `RecursiveCharacterTextSplitter.from_language(Language.PYTHON)`을 호출하면 Python 코드의 클래스, 함수, 메서드 경계를 존중하는 분리자 세트가 자동 설정됩니다. JavaScript, Go, Rust 등 다양한 언어를 지원하거든요.

> 🔥 **실무 팁**: `chunk_overlap`을 너무 크게 잡으면(30% 이상) 검색 시 동일한 정보가 여러 청크에서 중복 반환되어 오히려 LLM의 컨텍스트 윈도우를 낭비합니다. 10~20%가 최적이며, 특히 비용에 민감한 프로덕션 환경에서는 15%를 추천합니다.

## 핵심 정리

| 개념 | 설명 |
|------|------|
| CharacterTextSplitter | 단일 분리자(기본: `\n\n`)로만 분할. 간단하지만 `chunk_size` 초과 가능 |
| RecursiveCharacterTextSplitter | 분리자 계층(`\n\n` → `\n` → ` ` → `""`)을 재귀적으로 순회하며 분할. LangChain 권장 |
| 분리자 계층 | 큰 의미 단위(단락)부터 작은 단위(문자)로 순서대로 시도 |
| chunk_size | 청크의 최대 크기. 400~512자(토큰)가 범용적 시작점 |
| chunk_overlap | 인접 청크 간 공유 텍스트 양. chunk_size의 10~20%가 적정 |
| 한국어 커스터마이징 | `". "`, `"다. "` 등 한국어 문장 경계를 분리자에 추가 |
| 5 Levels of Splitting | Greg Kamradt의 프레임워크: 문자→재귀→문서별→시멘틱→에이전틱 |
