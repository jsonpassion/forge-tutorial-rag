# 청킹 전략 비교 실험과 최적화

> 동일한 문서에 서로 다른 청킹 전략을 적용하고, 검색 품질을 정량적으로 비교하여 최적의 전략을 선택하는 실전 가이드

이 섹션에서는 Chapter 4 전체를 관통하는 핵심 질문에 답합니다 — "그래서 어떤 청킹 전략을 써야 하나요?" **선수 지식**: [4.2 고정 크기 청킹과 재귀적 청킹](04-텍스트-청킹-전략-문서-분할과-최적화/02-고정-크기-청킹과-재귀적-청킹.md)의 `CharacterTextSplitter`/`RecursiveCharacterTextSplitter`, [4.3 문서 유형별 특화 청킹](04-텍스트-청킹-전략-문서-분할과-최적화/03-문서-유형별-특화-청킹.md)의 `MarkdownHeaderTextSplitter`와 2단계 분할, [4.4 시멘틱 청킹](04-텍스트-청킹-전략-문서-분할과-최적화/04-시멘틱-청킹-의미-기반-분할.md)의 `SemanticChunker`

**학습 목표**:
- 여러 청킹 전략의 검색 품질을 동일 조건에서 정량 비교할 수 있다
- 청크 크기별 성능 변화를 실험적으로 측정하고 시각화할 수 있다
- 문서 유형과 쿼리 패턴에 따른 최적 청킹 전략 선택 기준을 수립할 수 있다
- 프로덕션 환경에서 사용할 수 있는 청킹 평가 파이프라인을 구축할 수 있다

## 왜 알아야 할까?

RAG 시스템의 성능을 좌우하는 요인 중 청킹 전략은 가장 **비용 대비 효과가 큰 튜닝 포인트**입니다. 같은 임베딩 모델, 같은 벡터 DB, 같은 LLM을 사용하더라도 청킹 전략 하나만 바꿔도 검색 정확도가 20~30% 이상 달라질 수 있거든요.

문제는, "최고의 청킹 전략"은 존재하지 않는다는 겁니다. NVIDIA의 2024년 벤치마크에서 페이지 단위 청킹이 0.648 정확도로 1위를 차지했지만, 이건 PDF 기반 재무 보고서에서의 결과였습니다. 기술 블로그를 청킹한다면? 소스 코드를 청킹한다면? 결과가 완전히 달라지죠. 결국 **자신의 데이터와 쿼리 패턴에 맞는 전략을 실험으로 찾는 능력**이 가장 중요합니다.

이 섹션을 마치면, "느낌"이 아니라 **숫자**로 청킹 전략을 선택하는 엔지니어가 될 수 있습니다.

## 핵심 개념

### 개념 1: 청킹 평가 프레임워크 — 무엇을 측정할 것인가?

> 💡 **비유**: 청킹 평가는 요리 대회의 심사와 같습니다. 맛(검색 정확도)만 보는 게 아니라, 재료 활용도(토큰 효율성), 조리 시간(처리 속도), 플레이팅(청크 품질)까지 종합적으로 평가해야 하죠.

청킹 전략을 비교하려면 먼저 **무엇을 기준으로 비교할지** 정해야 합니다. Chroma Research에서 2024년 발표한 청킹 평가 프레임워크는 네 가지 핵심 메트릭을 제안합니다:

| 메트릭 | 설명 | 높을수록 |
|--------|------|----------|
| **Recall** | 정답에 필요한 토큰 중 검색된 청크에 포함된 비율 | 누락 없음 |
| **Precision** | 검색된 청크의 토큰 중 실제 정답에 필요한 비율 | 노이즈 없음 |
| **IoU** | 검색 토큰과 정답 토큰의 교집합/합집합 비율 | 효율적 |
| **처리 시간** | 청킹 + 임베딩에 걸리는 시간 | (낮을수록) |

여기서 **IoU(Intersection over Union)**가 특히 주목할 만한데요. 컴퓨터 비전에서 바운딩 박스의 정확도를 측정할 때 쓰던 메트릭을 텍스트 청킹에 적용한 겁니다. 청크를 "텍스트의 바운딩 박스"로 보는 발상이 참 영리하죠?

```python
# IoU 개념을 코드로 이해하기
def token_iou(retrieved_tokens: set, relevant_tokens: set) -> float:
    """텍스트 검색의 IoU(Intersection over Union) 계산"""
    intersection = retrieved_tokens & relevant_tokens  # 교집합: 정확히 맞은 토큰
    union = retrieved_tokens | relevant_tokens          # 합집합: 전체 관련 토큰
    return len(intersection) / len(union) if union else 0.0

# 예시: "RAG는 검색 증강 생성입니다"가 정답일 때
relevant = {"RAG", "는", "검색", "증강", "생성", "입니다"}
retrieved = {"RAG", "는", "검색", "증강", "생성", "입니다", "LLM", "의", "한계를"}  # 노이즈 포함

iou = token_iou(retrieved, relevant)
# IoU = 6/9 = 0.667 → 노이즈가 많아 효율이 떨어짐
```

### 개념 2: 비교 실험 설계 — 공정한 비교의 조건

> 💡 **비유**: 과학 실험에서 대조군과 실험군을 나누듯, 청킹 전략 비교에서도 **변인 통제**가 핵심입니다. 임베딩 모델, 검색 알고리즘, top-k 값은 고정하고, 오직 청킹 전략만 바꿔야 공정한 비교가 되죠.

공정한 비교 실험을 위한 설계 원칙은 다음과 같습니다:

1. **동일한 문서셋** 사용 — 전략마다 다른 문서를 쓰면 비교 불가
2. **동일한 임베딩 모델** — 모든 전략에 같은 임베딩 적용
3. **동일한 검색 파라미터** — top-k, 유사도 메트릭 통일
4. **동일한 평가 쿼리셋** — 미리 준비한 질문-정답 쌍으로 평가
5. **다양한 쿼리 유형** — 사실 기반(factoid), 분석적(analytical), 요약형(summary) 혼합

```python
from dataclasses import dataclass

@dataclass
class ChunkingExperiment:
    """청킹 비교 실험 설정"""
    # 통제 변인 (고정)
    embedding_model: str = "text-embedding-3-small"
    top_k: int = 5
    similarity_metric: str = "cosine"

    # 독립 변인 (변경)
    strategies: list = None  # 비교할 청킹 전략들
    chunk_sizes: list = None  # 비교할 청크 크기들

    # 종속 변인 (측정)
    metrics: list = None  # recall, precision, iou, time
```

### 개념 3: 청크 크기별 성능 곡선 — 스위트 스팟 찾기

> 💡 **비유**: 청크 크기 조정은 카메라 줌 조절과 비슷합니다. 너무 가까이 당기면(작은 청크) 디테일은 보이지만 전체 맥락을 잃고, 너무 멀리 빼면(큰 청크) 전체는 보이지만 필요한 부분을 정확히 찾기 어렵죠. 최적의 줌 레벨은 **무엇을 찍느냐**에 따라 달라집니다.

최근 연구들이 공통적으로 밝힌 청크 크기별 성능 패턴이 있습니다:

| 청크 크기 | 적합한 쿼리 유형 | 장점 | 단점 |
|-----------|-----------------|------|------|
| 64~128 토큰 | 사실 기반 짧은 질문 | 높은 검색 정밀도 | 맥락 부족 |
| 256~512 토큰 | 일반적인 QA | **균형 잡힌 성능** | — |
| 512~1024 토큰 | 분석적/맥락 의존 질문 | 풍부한 맥락 | 노이즈 증가 |
| 1024+ 토큰 | 요약, 종합 분석 | 최대 맥락 | 검색 정밀도 하락 |

Chroma Research의 실험 결과에 따르면, `RecursiveCharacterTextSplitter`를 **200~400 토큰, 오버랩 없음**으로 설정했을 때 단순함 대비 가장 좋은 성능을 보였습니다. 하지만 이건 영문 기준이고, 한국어에서는 형태소 특성을 고려해 약간의 오버랩(10~15%)을 주는 것이 경험적으로 더 좋은 결과를 줍니다.

### 개념 4: 도메인별 최적 전략 매트릭스

> 💡 **비유**: 옷장 정리 방법이 옷의 종류에 따라 다르듯(셔츠는 접어서, 코트는 걸어서, 양말은 짝지어서), 문서의 종류에 따라 최적의 청킹 전략도 다릅니다.

4개 세션에서 배운 전략들을 문서 유형별로 정리하면 다음과 같습니다:

| 문서 유형 | 1순위 전략 | 2순위 전략 | 권장 청크 크기 | 근거 |
|-----------|-----------|-----------|--------------|------|
| 기술 문서/API 레퍼런스 | `MarkdownHeaderTextSplitter` + 재귀 | `RecursiveCharacterTextSplitter` | 300~500자 | 헤더 구조가 의미 경계와 일치 |
| 뉴스/블로그 | `RecursiveCharacterTextSplitter` | `SemanticChunker` | 400~800자 | 비구조적 텍스트, 단락 단위 |
| 법률/계약서 | `RecursiveCharacterTextSplitter` | 조항 기반 커스텀 | 500~1000자 | 조항 번호가 자연 경계 |
| 소스 코드 | `from_language()` | 재귀 (커스텀 분리자) | 함수/클래스 단위 | 함수 경계 보존 필수 |
| 논문/학술 문서 | 2단계 (구조→크기) | `SemanticChunker` | 400~600자 | 섹션 구조 활용, 수식 보존 |
| FAQ/QA 데이터 | 질문-답변 쌍 단위 | 고정 크기 | 질문+답변 전체 | 의미 단위가 명확 |
| 한국어 일반 텍스트 | `RecursiveCharacterTextSplitter` (한국어 분리자) | `SemanticChunker` | 300~600자 | 문장 경계 + 조사 처리 |

> ⚠️ **흔한 오해**: "시멘틱 청킹이 항상 최고다"라고 생각하기 쉽지만, Chroma의 벤치마크에서 `RecursiveCharacterTextSplitter`(200~400 토큰)가 시멘틱 청킹과 비슷하거나 더 나은 결과를 보인 경우가 많았습니다. 시멘틱 청킹은 임베딩 비용이 추가로 들기 때문에, **성능 차이가 미미하다면 재귀적 분할이 더 경제적**입니다.

### 2단계: 샘플 문서와 평가 쿼리셋 준비

```python
# 비교 실험용 샘플 마크다운 문서
SAMPLE_DOCUMENT = """
# RAG 시스템 아키텍처 가이드

## 1. 개요

RAG(Retrieval-Augmented Generation)는 외부 지식 소스에서 관련 정보를 검색하여
LLM의 응답 품질을 높이는 기법입니다. 2020년 Facebook AI Research(현 Meta AI)의
Patrick Lewis 등이 제안한 이 방법은, LLM의 할루시네이션 문제를 크게 줄여줍니다.

RAG의 핵심 아이디어는 단순합니다: LLM이 모든 것을 "기억"할 필요 없이,
필요할 때 외부 데이터베이스에서 관련 정보를 "검색"하여 답변에 활용하는 것입니다.

## 2. 핵심 컴포넌트

### 2.1 문서 로더(Document Loader)

문서 로더는 PDF, 웹페이지, 데이터베이스 등 다양한 소스에서 텍스트를 추출합니다.
LangChain은 80개 이상의 문서 로더를 제공하며, Unstructured.io와의 통합으로
복잡한 레이아웃의 문서도 처리할 수 있습니다.

### 2.2 텍스트 스플리터(Text Splitter)

추출된 텍스트를 검색에 적합한 크기의 청크로 분할합니다.
청크 크기(chunk_size)와 오버랩(chunk_overlap)이 핵심 파라미터이며,
문서 유형에 따라 적절한 분할 전략을 선택해야 합니다.
일반적으로 256~512 토큰이 가장 좋은 성능을 보입니다.

### 2.3 임베딩 모델(Embedding Model)

텍스트 청크를 고차원 벡터로 변환합니다.
OpenAI의 text-embedding-3-small(1536차원)이 가성비가 뛰어나며,
오픈소스로는 sentence-transformers의 all-MiniLM-L6-v2가 널리 사용됩니다.
임베딩 모델의 토큰 제한(보통 512~8192 토큰)은 청크 크기 결정에 직접적인 영향을 줍니다.

### 2.4 벡터 데이터베이스(Vector Database)

임베딩 벡터를 저장하고 유사도 기반 검색을 수행합니다.
ChromaDB는 로컬 개발에 적합하고, Pinecone은 클라우드 관리형 서비스로
프로덕션 환경에서 널리 사용됩니다. FAISS는 Meta가 개발한
고성능 라이브러리로, 대규모 벡터 검색에 최적화되어 있습니다.

### 2.5 리트리버(Retriever)

사용자 쿼리를 임베딩하고 벡터 DB에서 유사한 청크를 검색합니다.
기본적인 유사도 검색 외에도 MMR(Maximal Marginal Relevance),
상위 k 검색, 메타데이터 필터링 등 다양한 검색 전략이 있습니다.

## 3. 파이프라인 흐름

전체 RAG 파이프라인은 크게 두 단계로 나뉩니다:

**인덱싱 단계**: 문서 로드 → 텍스트 분할 → 임베딩 생성 → 벡터 DB 저장
**검색-생성 단계**: 쿼리 임베딩 → 유사 청크 검색 → 프롬프트 구성 → LLM 생성

인덱싱은 오프라인으로 한 번만 수행하고, 검색-생성은 매 쿼리마다 실행됩니다.
따라서 인덱싱 단계의 최적화(청킹 전략, 임베딩 모델 선택)가
전체 시스템 성능에 큰 영향을 미칩니다.

## 4. 성능 최적화

### 4.1 청킹 최적화

청크 크기가 너무 작으면 맥락이 부족하고, 너무 크면 노이즈가 증가합니다.
```

```python
def apply_chunking_strategies(document: str) -> dict[str, list[Document]]:
    """동일 문서에 여러 청킹 전략을 적용하여 결과 비교"""

    results = {}

    # 전략 1: 고정 크기 (CharacterTextSplitter)
    fixed_splitter = CharacterTextSplitter(
        separator="\n",
        chunk_size=500,
        chunk_overlap=50,
    )
    results["fixed_500"] = fixed_splitter.create_documents([document])

    # 전략 2: 재귀적 분할 — 여러 크기로 실험
    for size in [200, 400, 600, 800]:
        recursive_splitter = RecursiveCharacterTextSplitter(
            chunk_size=size,
            chunk_overlap=int(size * 0.1),  # 10% 오버랩
            separators=["\n\n", "\n", ". ", " ", ""],  # 한국어 최적화
        )
        results[f"recursive_{size}"] = recursive_splitter.create_documents(
            [document]
        )

    # 전략 3: 마크다운 구조 기반 (2단계 분할)
    md_splitter = MarkdownHeaderTextSplitter(
        headers_to_split_on=[
            ("#", "h1"),
            ("##", "h2"),
            ("###", "h3"),
        ]
    )
    md_chunks = md_splitter.split_text(document)
    # 2단계: 큰 청크를 추가 분할
    recursive_for_md = RecursiveCharacterTextSplitter(
        chunk_size=400,
        chunk_overlap=40,
    )
    results["markdown_2stage"] = recursive_for_md.split_documents(md_chunks)

    # 전략 4: 시멘틱 청킹
    embeddings = OpenAIEmbeddings(model="text-embedding-3-small")
    semantic_splitter = SemanticChunker(
        embeddings,
        breakpoint_threshold_type="percentile",
        breakpoint_threshold_amount=75,
    )
    results["semantic_p75"] = semantic_splitter.create_documents([document])

    return results
```

### 4단계: 검색 품질 평가 엔진

```python
import chromadb
from chromadb.utils.embedding_functions import OpenAIEmbeddingFunction

@dataclass
class EvalResult:
    """단일 전략의 평가 결과"""
    strategy_name: str
    num_chunks: int
    avg_chunk_size: float
    recall: float           # 정답 키워드 중 검색된 비율
    precision: float        # 검색 결과 중 관련 청크 비율
    avg_query_time: float   # 평균 검색 시간 (초)
    chunking_time: float    # 청킹 소요 시간 (초)

def evaluate_strategy(
    strategy_name: str,
    chunks: list[Document],
    queries: list[dict],
    top_k: int = 3,
) -> EvalResult:
    """단일 청킹 전략의 검색 품질을 평가"""

    # ChromaDB 인메모리 컬렉션 생성
    client = chromadb.Client()
    embedding_fn = OpenAIEmbeddingFunction(
        api_key=os.environ["OPENAI_API_KEY"],
        model_name="text-embedding-3-small",
    )

    # 기존 컬렉션 삭제 후 새로 생성
    collection_name = f"eval_{strategy_name}"
    try:
        client.delete_collection(collection_name)
    except Exception:
        pass
    collection = client.create_collection(
        name=collection_name,
        embedding_function=embedding_fn,
    )

    # 청크 인덱싱
    chunk_texts = [doc.page_content for doc in chunks]
    chunk_ids = [f"chunk_{i}" for i in range(len(chunks))]
    collection.add(documents=chunk_texts, ids=chunk_ids)

    # 쿼리별 평가
    total_recall = 0.0
    total_precision = 0.0
    total_query_time = 0.0

    for q in queries:
        start = time.time()
        results = collection.query(
            query_texts=[q["query"]],
            n_results=min(top_k, len(chunks)),
        )
        query_time = time.time() - start
        total_query_time += query_time

        # 검색된 텍스트에서 키워드 매칭으로 recall 계산
        retrieved_text = " ".join(results["documents"][0])
        found_keywords = [
            kw for kw in q["expected_keywords"]
            if kw in retrieved_text
        ]
        recall = len(found_keywords) / len(q["expected_keywords"])
        total_recall += recall

        # 관련 청크 비율 (최소 1개 키워드 포함 시 관련으로 판정)
        relevant_count = sum(
            1 for doc in results["documents"][0]
            if any(kw in doc for kw in q["expected_keywords"])
        )
        precision = relevant_count / len(results["documents"][0])
        total_precision += precision

    num_queries = len(queries)
    avg_chunk_size = sum(len(doc.page_content) for doc in chunks) / len(chunks)

    return EvalResult(
        strategy_name=strategy_name,
        num_chunks=len(chunks),
        avg_chunk_size=round(avg_chunk_size, 1),
        recall=round(total_recall / num_queries, 3),
        precision=round(total_precision / num_queries, 3),
        avg_query_time=round(total_query_time / num_queries, 4),
        chunking_time=0.0,  # 별도 측정
    )
```

```python
from pathlib import Path

class ProductionChunkingPipeline:
    """프로덕션 환경을 위한 청킹 파이프라인

    자동 전략 선택, 품질 검증, 메타데이터 보강을 포함합니다.
    """

    # 문서 유형별 기본 전략 매핑
    STRATEGY_MAP = {
        ".md": "markdown_2stage",
        ".html": "html_2stage",
        ".py": "code_language",
        ".js": "code_language",
        ".txt": "recursive",
        ".pdf": "recursive",
    }

    def __init__(
        self,
        default_chunk_size: int = 400,
        default_overlap: int = 40,
        min_chunk_size: int = 50,
    ):
        self.default_chunk_size = default_chunk_size
        self.default_overlap = default_overlap
        self.min_chunk_size = min_chunk_size  # 이보다 작은 청크는 필터링

    def chunk_document(
        self,
        text: str,
        file_ext: str = ".txt",
        metadata: dict | None = None,
    ) -> list[Document]:
        """문서 유형에 맞는 전략으로 자동 청킹"""

        strategy = self.STRATEGY_MAP.get(file_ext, "recursive")
        chunks = self._apply_strategy(text, strategy)

        # 품질 필터: 너무 짧은 청크 제거
        chunks = [c for c in chunks if len(c.page_content) >= self.min_chunk_size]

        # 메타데이터 보강
        if metadata:
            for chunk in chunks:
                chunk.metadata.update(metadata)

        # 청크 인덱스 추가
        for i, chunk in enumerate(chunks):
            chunk.metadata["chunk_index"] = i
            chunk.metadata["total_chunks"] = len(chunks)
            chunk.metadata["strategy"] = strategy

        return chunks

    def _apply_strategy(self, text: str, strategy: str) -> list[Document]:
        """전략별 청킹 실행"""

        if strategy == "markdown_2stage":
            md_splitter = MarkdownHeaderTextSplitter(
                headers_to_split_on=[("#", "h1"), ("##", "h2"), ("###", "h3")]
            )
            md_chunks = md_splitter.split_text(text)
            recursive = RecursiveCharacterTextSplitter(
                chunk_size=self.default_chunk_size,
                chunk_overlap=self.default_overlap,
            )
            return recursive.split_documents(md_chunks)

        elif strategy == "code_language":
            # Python 기본, 필요시 확장
            splitter = RecursiveCharacterTextSplitter.from_language(
                language="python",
                chunk_size=self.default_chunk_size * 2,  # 코드는 더 크게
                chunk_overlap=0,  # 함수 경계에서 분할하므로 오버랩 불필요
            )
            return splitter.create_documents([text])

        else:  # recursive (기본)
            splitter = RecursiveCharacterTextSplitter(
                chunk_size=self.default_chunk_size,
                chunk_overlap=self.default_overlap,
                separators=["\n\n", "\n", ". ", ", ", " ", ""],
            )
            return splitter.create_documents([text])

    def validate_chunks(self, chunks: list[Document]) -> dict:
        """청크 품질 검증 리포트 생성"""
        sizes = [len(c.page_content) for c in chunks]
        return {
            "total_chunks": len(chunks),
            "avg_size": round(sum(sizes) / len(sizes), 1) if sizes else 0,
            "min_size": min(sizes) if sizes else 0,
            "max_size": max(sizes) if sizes else 0,
            "too_small": sum(1 for s in sizes if s < self.min_chunk_size),
            "too_large": sum(1 for s in sizes if s > self.default_chunk_size * 3),
        }
```

### 청킹 평가의 역사 — 컴퓨터 비전에서 NLP로

"텍스트 청킹의 품질을 어떻게 측정할 것인가?"는 생각보다 최근까지 명확한 답이 없던 질문이었습니다. RAG가 폭발적으로 성장하던 2023년에도, 대부분의 개발자들은 청킹 결과를 "눈으로 보고" 괜찮은지 판단했죠.

2024년 7월, Chroma의 Brandon Smith와 Anton Troynikov가 발표한 기술 보고서 *"Evaluating Chunking Strategies for Retrieval"*이 이 상황을 바꿔놓았습니다. 핵심 아이디어는 **컴퓨터 비전의 IoU 메트릭을 텍스트 검색에 차용**한 것이었는데요.

컴퓨터 비전에서 IoU는 객체 탐지 모델이 예측한 바운딩 박스가 실제 객체를 얼마나 정확히 감싸는지 측정합니다. 이를 텍스트에 적용하면, "검색된 청크가 정답에 필요한 토큰을 얼마나 정확히 포함하는가?"를 측정할 수 있습니다. 검색된 청크에 불필요한 토큰이 많으면(바운딩 박스가 너무 크면) IoU가 떨어지고, 필요한 토큰이 빠져 있어도(바운딩 박스가 너무 작아도) IoU가 떨어지죠.

이 보고서의 또 다른 흥미로운 결과는, **오버랩을 넣지 않은 `RecursiveCharacterTextSplitter`(200~400 토큰)가 대부분의 시멘틱 청킹보다 IoU에서 우세했다**는 점입니다. 이는 "고급 기법이 항상 좋다"는 통념을 데이터로 깨뜨린 사례로, 실무자들에게 큰 반향을 일으켰습니다.

### NVIDIA의 대규모 벤치마크 (2024)

같은 시기, NVIDIA도 독자적인 청킹 벤치마크를 발표했습니다. 7가지 청킹 전략을 5개 데이터셋(767개 PDF, 600개 이상의 질문)으로 평가한 대규모 실험이었는데요. 결과는 놀라웠습니다 — 가장 "단순한" 전략인 **페이지 단위 청킹**이 0.648 정확도와 가장 낮은 분산(0.107)으로 1위를 차지한 겁니다.

물론 이 결과는 "페이지"라는 개념이 명확한 PDF 문서에 한정된 것이지만, **단순한 전략을 먼저 시도하고, 데이터로 개선점을 찾아가라**는 교훈은 보편적으로 적용됩니다.

## 흔한 오해와 팁

> ⚠️ **흔한 오해**: "청크 크기가 작을수록 검색이 정확하다"고 생각하기 쉽지만, 지나치게 작은 청크(100자 미만)는 오히려 **맥락 부족**으로 LLM이 좋은 답변을 생성하지 못합니다. 검색 정밀도와 생성 품질은 서로 다른 축이며, 청크 크기는 이 둘의 균형점에서 결정해야 합니다.

> 💡 **알고 계셨나요?**: Chroma Research에 따르면, 오버랩(chunk_overlap)을 0으로 설정해도 `RecursiveCharacterTextSplitter`의 IoU 성능이 거의 떨어지지 않았습니다. 오버랩은 직관적으로 "안전장치"처럼 느껴지지만, 실제로는 **인덱싱 비용만 늘리고 성능 향상은 미미**한 경우가 많습니다. 먼저 오버랩 없이 테스트하고, 성능이 부족할 때만 추가하는 것이 효율적입니다.

> 🔥 **실무 팁**: 청킹 전략을 선택할 때 이 순서를 따르세요 — ① `RecursiveCharacterTextSplitter(chunk_size=400, overlap=0)`로 **베이스라인**을 먼저 잡는다 ② 문서에 구조가 있다면(마크다운, HTML) **구조 인식 + 재귀 2단계**를 시도한다 ③ 성능 차이가 유의미한 경우에만 시멘틱 청킹을 고려한다. 80%의 프로젝트에서 ①이면 충분하고, 15%에서 ②가 필요하며, 시멘틱 청킹이 정말 필요한 경우는 5% 미만입니다.

> 🔥 **실무 팁**: 청킹 실험 결과를 반드시 **버전 관리**하세요. 실험 설정(전략, 파라미터), 평가 쿼리셋, 메트릭 결과를 JSON이나 CSV로 저장해 두면, 나중에 문서가 추가되거나 요구사항이 바뀌었을 때 과거 실험과 비교할 수 있습니다. MLflow나 Weights & Biases 같은 실험 추적 도구를 활용하면 더 체계적으로 관리할 수 있습니다.

## 핵심 정리

| 개념 | 설명 |
|------|------|
| 청킹 평가 메트릭 | Recall, Precision, IoU로 검색 품질을 정량 측정. IoU는 토큰 수준의 효율성까지 반영 |
| 공정한 비교 설계 | 임베딩 모델·검색 파라미터·평가 쿼리셋을 통제하고, 청킹 전략만 변경 |
| 청크 크기 스위트 스팟 | 일반적으로 256~512 토큰이 균형 잡힌 성능. 쿼리 유형에 따라 조정 |
| 도메인별 전략 선택 | 구조적 문서는 2단계 분할, 비구조적 문서는 재귀적 분할이 기본 |
| 프로덕션 파이프라인 | 자동 전략 선택 + 품질 검증 + 메타데이터 보강으로 운영 안정성 확보 |
| 베이스라인 우선 원칙 | `RecursiveCharacterTextSplitter(400, 0)`부터 시작, 데이터로 개선점 탐색 |
