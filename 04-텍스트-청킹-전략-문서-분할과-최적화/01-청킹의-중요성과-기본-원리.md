# 청킹의 중요성과 기본 원리

> 긴 문서를 작은 조각으로 나누는 기술이 RAG 검색 품질을 좌우합니다

## 개요

이 섹션에서는 RAG 파이프라인에서 텍스트 청킹(Chunking)이 왜 필수적인지, 그리고 청크 크기와 오버랩이 검색 품질에 어떤 영향을 미치는지 알아봅니다. 좋은 청크가 갖추어야 할 조건을 이해하고, 직접 코드로 확인하면서 기본 원리를 체득하는 것이 목표입니다.

**선수 지식**: [Session 3.1: 문서 로딩의 기본 구조](../ch03/3.1)에서 정의한 `Document` 객체와 `page_content` 개념, 문서 로더 사용법
**학습 목표**:
- 청킹이 필요한 근본적인 이유(임베딩 모델 토큰 제한, 검색 정밀도)를 설명할 수 있다
- 청크 크기와 검색 품질 사이의 트레이드오프를 이해한다
- 오버랩의 역할과 적절한 설정 방법을 안다
- 좋은 청크의 조건(의미적 완결성, 적절한 길이)을 정의할 수 있다

## 왜 알아야 할까?

앞서 Ch3에서 다양한 소스로부터 문서를 로드하는 방법을 배웠는데요. 그런데 로드한 문서를 그대로 임베딩하고 검색하면 어떤 일이 벌어질까요?

가장 흔한 문제는 **"아무것도 안 찾아지거나, 엉뚱한 것이 찾아지는"** 상황입니다. 50페이지짜리 PDF를 통째로 임베딩하면 그 벡터는 문서 전체의 "평균적인 의미"를 담게 되어, 구체적인 질문에 대한 정확한 답을 찾기 어렵습니다. 반대로 문장 하나하나를 개별 청크로 만들면, 맥락이 사라져서 "이것"이 무엇을 가리키는지도 알 수 없게 되죠.

실제로 청킹 전략 하나만 바꿔도 RAG 시스템의 정확도가 65%에서 92%까지 뛸 수 있다는 벤치마크 결과가 있습니다. 청킹은 RAG 파이프라인에서 **가장 적은 비용으로 가장 큰 성능 향상**을 이끌어낼 수 있는 단계입니다.

## 핵심 개념

### 개념 1: 청킹이란 무엇인가

> 💡 **비유**: 도서관에서 책 한 권을 통째로 서가에 꽂는다고 상상해보세요. 누군가 "3장에 나오는 양자역학의 불확정성 원리에 대해 알려주세요"라고 물으면, 책 전체를 건네줄 수밖에 없습니다. 하지만 책을 장·절·문단 단위로 분리한 색인 카드가 있다면? 정확히 해당 문단만 꺼내줄 수 있죠. 청킹은 바로 이 **색인 카드를 만드는 작업**입니다.

청킹(Chunking)은 긴 문서를 **검색과 처리에 적합한 작은 텍스트 단위**로 분할하는 과정입니다. RAG 파이프라인에서 청킹이 필요한 이유는 크게 두 가지입니다.

**1. 임베딩 모델의 토큰 제한**

임베딩 모델은 입력할 수 있는 텍스트 길이에 상한이 있습니다. 예를 들어 OpenAI의 `text-embedding-3-small` 모델은 최대 8,191 토큰까지 처리할 수 있는데요, 이 한계를 넘는 텍스트는 아예 임베딩할 수 없습니다. 하지만 한계 내라고 해서 긴 텍스트를 통째로 넣는 것이 좋은 것도 아닙니다.

**2. 검색 정밀도(Retrieval Precision)**

긴 텍스트를 하나의 벡터로 압축하면 세부적인 의미가 희석됩니다. "서울의 평균 기온은?"이라는 질문에 대해, 한국 지리 전체를 담은 10,000자 청크와 서울 기후만 다루는 300자 청크가 있다면, 후자가 훨씬 정확하게 매칭되겠죠.

```python
# 청킹 없이 문서 전체를 임베딩하는 (나쁜) 예시
from langchain_community.document_loaders import TextLoader

loader = TextLoader("my_document.txt")
docs = loader.load()  # 문서 1개 = 매우 긴 텍스트

# docs[0].page_content가 수만 자라면?
# → 임베딩이 의미를 제대로 포착하지 못함
# → 검색 시 관련 없는 내용까지 함께 반환됨
```

### 개념 2: 청크 크기와 검색 품질의 관계

> 💡 **비유**: 피자를 자른다고 생각해보세요. 피자를 2조각으로 자르면(큰 청크) 한 조각에 다양한 토핑이 섞여 있어서 "페퍼로니만 있는 조각"을 고를 수 없습니다. 반대로 100조각으로 잘게 자르면(작은 청크) 각 조각이 너무 작아서 토핑 하나도 제대로 올라가 있지 않죠. **적당한 크기**로 자르는 것이 핵심입니다.

청크 크기는 RAG 성능에 직접적인 영향을 미칩니다. 이 트레이드오프를 표로 정리하면:

| 구분 | 작은 청크 (128~256 토큰) | 큰 청크 (512~1024 토큰) |
|------|-------------------------|------------------------|
| **장점** | 높은 검색 정밀도, 정확한 매칭 | 풍부한 맥락, 답변 생성에 유리 |
| **단점** | 맥락 부족, 대명사 참조 끊김 | 의미 희석, 불필요한 정보 포함 |
| **적합한 질문** | 사실 기반(팩트) 질문 | 분석·요약형 질문 |

실무에서 가장 많이 사용되는 기본값은 **400~512 토큰**입니다. 여러 벤치마크에서 이 범위가 85~90%의 리콜(Recall)을 달성하면서도 계산 부담이 적은 것으로 나타났거든요.

```run:python
# 청크 크기에 따른 분할 결과 비교
text = """인공지능(AI)은 인간의 학습, 추론, 지각 능력을 컴퓨터로 구현한 기술입니다.
머신러닝은 AI의 하위 분야로, 데이터에서 패턴을 학습합니다.
딥러닝은 머신러닝의 하위 분야로, 인공 신경망을 사용합니다.
자연어 처리(NLP)는 컴퓨터가 인간의 언어를 이해하고 생성하는 기술입니다.
RAG는 검색과 생성을 결합하여 LLM의 한계를 극복하는 방법입니다."""

# 작은 청크 (50자)
small_chunks = [text[i:i+50] for i in range(0, len(text), 50)]
print(f"=== 작은 청크 (50자) → {len(small_chunks)}개 ===")
for i, chunk in enumerate(small_chunks[:3]):
    print(f"  청크 {i}: '{chunk.strip()}'")

print()

# 큰 청크 (150자)
big_chunks = [text[i:i+150] for i in range(0, len(text), 150)]
print(f"=== 큰 청크 (150자) → {len(big_chunks)}개 ===")
for i, chunk in enumerate(big_chunks[:2]):
    print(f"  청크 {i}: '{chunk.strip()}'")
```

```output

  청크 1: '다.

딥러닝은'
  청크 2: 'AI의 하위 분야로, 인공 신경망을 사용합니다.
자연어 처리(NLP)는 컴'

=== 큰 청크 (150자) → 2개 ===
  청크 0: '인공지능(AI)은 인간의 학습, 추론, 지각 능력을 컴퓨터로 구현한 기술입니다.
머신러닝은 AI의 하위 분야로, 데이터에서 패턴을 학습합니다.
딥러닝은 머신러닝의 하위 분야로, 인공 신경망을 사용합니다.
자연어 처리(NLP)는 컴퓨터가 인간의 언어를'
  청크 1: '이해하고 생성하는 기술입니다.
RAG는 검색과 생성을 결합하여 LLM의 한계를 극복하는 방법입니다.'
```

작은 청크에서는 "다."로 시작하거나 문장이 중간에 잘리는 것을 볼 수 있습니다. 반면 큰 청크는 여러 주제가 하나의 청크에 섞여 있죠. 이런 문제를 해결하는 첫 번째 도구가 바로 **오버랩**입니다.

### 개념 3: 오버랩의 역할

> 💡 **비유**: 퍼즐 조각을 만들 때, 각 조각의 가장자리가 이웃 조각과 살짝 겹치도록 설계하면 조각들을 맞추기가 훨씬 쉬워지죠. 오버랩은 청크 사이의 **겹침 영역**을 만들어서, 경계에서 잘린 문맥을 이어주는 역할을 합니다.

오버랩(Overlap)은 인접한 청크 사이에 일부 텍스트를 공유하는 기법입니다. 예를 들어 청크 크기가 500자이고 오버랩이 100자라면, 청크 1의 마지막 100자가 청크 2의 처음 100자와 동일합니다.

왜 필요할까요? 문장이 청크 경계에서 잘릴 때, 오버랩이 없으면 앞 청크와 뒤 청크 모두 불완전한 정보를 갖게 됩니다. 오버랩이 있으면 경계 부근의 문장이 양쪽 청크에 모두 포함되어 **의미의 연속성**이 보존됩니다.

```
오버랩 없이 분할:
[청크1: "... 딥러닝은 머신러닝의"] [청크2: "하위 분야로, 신경망을 ..."]
→ 청크1만으로는 "딥러닝이 뭔지" 불완전
→ 청크2만으로는 "뭐의 하위 분야인지" 불완전

오버랩 있게 분할:
[청크1: "... 딥러닝은 머신러닝의 하위 분야로"] [청크2: "딥러닝은 머신러닝의 하위 분야로, 신경망을 ..."]
→ 양쪽 모두 완전한 문장을 포함
```

업계에서 권장하는 오버랩 크기는 **청크 크기의 10~20%** 입니다. 500토큰 청크라면 50~100토큰 정도가 적절하죠. 너무 크면 중복 데이터가 많아져 저장 공간과 검색 비용이 늘어나고, 너무 작으면 경계에서의 문맥 손실 방지 효과가 약해집니다.

### 개념 4: 좋은 청크의 조건

그렇다면 "좋은 청크"란 어떤 것일까요? 핵심 조건은 세 가지입니다.

**1. 의미적 완결성 (Semantic Coherence)**
하나의 청크가 하나의 주제나 아이디어를 완결적으로 담고 있어야 합니다. "딥러닝은"으로 끝나는 청크는 좋은 청크가 아닙니다.

**2. 적절한 길이**
임베딩 모델이 의미를 제대로 포착할 수 있는 길이여야 합니다. 너무 짧으면(한두 문장) 맥락이 부족하고, 너무 길면(수천 자) 의미가 희석됩니다.

**3. 메타데이터 보존**
청크가 원본 문서의 어디에서 왔는지(출처, 페이지 번호, 섹션 제목 등) 추적할 수 있어야 합니다. 이 정보는 나중에 검색 결과의 출처를 사용자에게 보여줄 때 필수적이거든요.

```run:python
# 좋은 청크 vs 나쁜 청크 비교
bad_chunk_1 = "딥러닝은 머신러닝의"  # 문장 불완전
bad_chunk_2 = """인공지능의 역사는 1950년대로 거슬러 올라간다. 앨런 튜링은 기계가 
생각할 수 있는지 물었다. 한편 오늘날 자연어 처리 기술은 번역, 요약, 질의응답 등에 
활용된다. 또한 컴퓨터 비전은 자율주행차에 핵심이다. 강화학습은 게임 AI에서 
탁월한 성과를 보였다."""  # 여러 주제가 혼재

good_chunk = """딥러닝은 머신러닝의 하위 분야로, 인공 신경망을 사용하여 
데이터에서 복잡한 패턴을 학습합니다. 여러 층의 신경망을 쌓아 
'깊은(Deep)' 구조를 만들며, 이미지 인식, 자연어 처리 등에서 
뛰어난 성과를 보여주고 있습니다."""  # 하나의 주제, 완결된 의미

print("=== 나쁜 청크 예시 1 ===")
print(f"길이: {len(bad_chunk_1)}자")
print(f"내용: '{bad_chunk_1}'")
print(f"문제: 문장이 잘려서 의미 불완전\n")

print("=== 나쁜 청크 예시 2 ===")
print(f"길이: {len(bad_chunk_2)}자")
print(f"문제: AI 역사, NLP, 컴퓨터 비전, 강화학습 등 여러 주제 혼재\n")

print("=== 좋은 청크 예시 ===")
print(f"길이: {len(good_chunk)}자")
print(f"문제: 없음 — '딥러닝'이라는 하나의 주제를 완결적으로 설명")
```

```output
=== 나쁜 청크 예시 1 ===
길이: 10자
내용: '딥러닝은 머신러닝의'
문제: 문장이 잘려서 의미 불완전

=== 나쁜 청크 예시 2 ===
길이: 142자
문제: AI 역사, NLP, 컴퓨터 비전, 강화학습 등 여러 주제 혼재

=== 좋은 청크 예시 ===
길이: 110자
문제: 없음 — '딥러닝'이라는 하나의 주제를 완결적으로 설명
```

## 실습: 직접 해보기

이제 LangChain의 텍스트 스플리터를 사용해서, 청크 크기와 오버랩이 실제 분할 결과에 어떤 영향을 미치는지 직접 실험해봅시다.

```python
# 필요한 패키지 설치
# pip install langchain-text-splitters
```

```run:python
from langchain_text_splitters import (
    CharacterTextSplitter,
    RecursiveCharacterTextSplitter,
)

# 실습용 한국어 텍스트 (RAG에 대한 설명)
sample_text = """RAG(Retrieval-Augmented Generation)는 검색 증강 생성이라고 번역되며, 
외부 지식 소스에서 관련 정보를 검색하여 LLM의 응답 생성을 보강하는 기법입니다.

RAG가 등장하기 전에는 LLM이 학습 데이터에만 의존했기 때문에, 최신 정보나 특정 도메인의 
전문 지식에 대해 정확한 답변을 하기 어려웠습니다. 또한 학습 데이터에 없는 내용을 
마치 사실인 것처럼 생성하는 할루시네이션(Hallucination) 문제도 심각했습니다.

RAG는 이 문제를 해결하기 위해 검색(Retrieval)과 생성(Generation)을 결합합니다.
사용자의 질문이 들어오면, 먼저 벡터 데이터베이스에서 관련 문서를 검색하고,
검색된 문서를 컨텍스트로 LLM에 전달하여 답변을 생성합니다.

이 과정에서 핵심적인 단계가 바로 청킹(Chunking)입니다.
원본 문서를 적절한 크기의 청크로 나누어야 벡터 검색이 효과적으로 작동합니다.
청크가 너무 크면 검색 정밀도가 떨어지고, 너무 작으면 맥락이 손실됩니다."""

# ========================================
# 실험 1: CharacterTextSplitter (단순 분리자 기반)
# ========================================
char_splitter = CharacterTextSplitter(
    separator="\n\n",        # 빈 줄(문단) 기준으로 분할
```

## 더 깊이 알아보기

### 청킹의 뿌리: 정보 검색(IR)에서 RAG까지

"텍스트를 잘게 나눠서 검색한다"는 아이디어는 사실 RAG보다 훨씬 오래된 개념입니다. 1970~80년대 정보 검색(Information Retrieval) 분야에서 이미 **문서를 단락(passage) 단위로 인덱싱하면 검색 성능이 올라간다**는 연구가 활발했거든요. 당시에는 TF-IDF 같은 키워드 기반 검색이 주류였는데, 문서 전체보다 단락 단위가 더 정확하다는 사실은 이미 알려져 있었습니다.

이 아이디어가 현대 RAG에서 부활한 것은 2020년 Facebook AI Research(현 Meta AI)의 논문 *"Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks"*에서입니다. 이 논문에서 DPR(Dense Passage Retrieval)을 사용해 Wikipedia의 **100단어 단위 패시지**를 검색하고 생성 모델에 전달하는 아키텍처를 제안했죠. 여기서 "100단어"라는 청크 크기는 당시 실험에서 경험적으로 결정된 것이었는데, 놀랍게도 이 값이 현재까지도 많은 RAG 시스템의 기본 설정에 영향을 미치고 있습니다.

### NAACL 2025의 도발적 발견

흥미로운 최근 연구도 있습니다. NAACL 2025 Findings에 발표된 논문에서는 "복잡한 시멘틱 청킹이 정말 고정 크기 청킹보다 나은가?"를 체계적으로 검증했는데요, 놀랍게도 **고정 200단어 청킹이 시멘틱 청킹과 비슷하거나 더 나은 성능**을 보였다는 결론을 내렸습니다. 물론 이것이 "시멘틱 청킹이 쓸모없다"는 뜻은 아닙니다. 다만 **기본기가 탄탄한 단순한 방법을 먼저 시도**하고, 필요할 때 복잡한 기법으로 넘어가야 한다는 교훈을 줍니다.

## 흔한 오해와 팁

> ⚠️ **흔한 오해**: "청크 크기는 작을수록 좋다"
> 직관적으로는 작은 청크가 더 정확할 것 같지만, 지나치게 작은 청크(50토큰 미만)는 오히려 성능을 떨어뜨립니다. 맥락이 부족해서 "이것", "그 결과" 같은 대명사 참조가 끊기고, 임베딩이 의미를 충분히 포착하지 못하기 때문입니다. 128토큰 이상을 유지하는 것이 일반적인 가이드라인입니다.

> 💡 **알고 계셨나요?**: 원본 RAG 논문(2020)에서는 Wikipedia 전체를 약 2,100만 개의 100단어 패시지로 분할했습니다. 이 단순한 고정 크기 분할만으로도 당시 최고 수준의 QA 성능을 달성했죠. "복잡한 청킹 전략부터 찾지 말고, 기본부터 시작하라"는 교훈을 주는 사례입니다.

> 🔥 **실무 팁**: 청크 크기를 정할 때 가장 확실한 방법은 **실제 사용자 질문 20~30개로 A/B 테스트**를 하는 것입니다. 256, 512, 1024 토큰 등 2~3가지 크기로 실험하고, 검색된 청크가 질문에 얼마나 정확하게 답하는지 사람이 직접 평가하세요. 이론적 최적값보다 데이터에 맞는 최적값이 항상 다릅니다.

> 🔥 **실무 팁**: 오버랩을 무조건 설정할 필요는 없습니다. 2026년 1월의 한 체계적 분석에서는 SPLADE 검색기와 함께 사용했을 때 오버랩이 측정 가능한 이점을 제공하지 못하고 인덱싱 비용만 증가시켰다고 보고했습니다. 자신의 데이터와 검색기 조합에서 실제로 효과가 있는지 테스트해보세요.

## 핵심 정리

| 개념 | 설명 |
|------|------|
| 청킹(Chunking) | 긴 문서를 검색에 적합한 작은 텍스트 단위로 분할하는 과정 |
| 청킹이 필요한 이유 | 임베딩 모델의 토큰 제한 + 검색 정밀도 향상 |
| 작은 청크 (128~256 토큰) | 높은 정밀도, 낮은 맥락. 팩트 질문에 적합 |
| 큰 청크 (512~1024 토큰) | 풍부한 맥락, 의미 희석 가능. 분석형 질문에 적합 |
| 오버랩(Overlap) | 인접 청크 간 겹침 영역. 경계에서의 문맥 손실 방지 (보통 10~20%) |
| 좋은 청크의 조건 | 의미적 완결성 + 적절한 길이 + 메타데이터 보존 |
| 권장 기본값 | chunk_size=400~512, overlap=50~100 (15% 내외) |
| RecursiveCharacterTextSplitter | LangChain의 권장 범용 텍스트 스플리터. 문단 → 줄 → 단어 순으로 분할 시도 |
