# 벡터 데이터베이스란 — 왜 필요한가

> 임베딩 벡터를 저장하고 "의미적으로 비슷한 것"을 빠르게 찾아주는 특수 데이터베이스, 벡터 데이터베이스의 세계로 들어갑니다.

## 개요

이 섹션에서는 벡터 데이터베이스가 무엇이고, 왜 전통적인 데이터베이스로는 부족한지를 이해합니다. 수백만 개의 벡터 중에서 가장 비슷한 것을 빠르게 찾아내는 ANN(Approximate Nearest Neighbor) 알고리즘의 원리를 살펴보고, 대표적인 인덱싱 기법인 HNSW와 IVF를 개념적으로 이해합니다. 마지막으로 현재 벡터 데이터베이스 시장의 주요 플레이어들을 비교해봅니다.

**선수 지식**: [Ch5: 임베딩 모델 이해](ch05)에서 배운 임베딩의 개념, 벡터 공간에서의 코사인 유사도(Cosine Similarity) 계산 방법
**학습 목표**:
- 전통적 데이터베이스와 벡터 데이터베이스의 근본적 차이를 설명할 수 있다
- ANN 검색이 왜 필요한지, 정확한 검색과 어떤 트레이드오프가 있는지 이해한다
- HNSW와 IVF 인덱싱 알고리즘의 핵심 아이디어를 비유로 설명할 수 있다
- 주요 벡터 데이터베이스(ChromaDB, FAISS, Pinecone, Qdrant)의 특징과 용도를 구분할 수 있다

## 왜 알아야 할까?

앞서 Ch5에서 텍스트를 임베딩 벡터로 변환하는 방법을 배웠습니다. 그런데 벡터를 만들기만 하면 뭘 할 수 있을까요? "서울 맛집 추천해줘"라는 질문이 들어왔을 때, 수십만 개의 문서 벡터 중에서 가장 관련 있는 문서를 **밀리초 단위**로 찾아내야 합니다.

Python 리스트에 벡터를 저장하고 하나씩 비교하면? 문서 1,000개일 때는 괜찮지만, 100만 개가 되면 매 쿼리마다 몇 초씩 걸립니다. 1억 개라면? 아예 실용적이지 않죠.

벡터 데이터베이스는 RAG 파이프라인의 **중추 신경**입니다. 검색 품질과 속도가 최종 LLM 응답의 품질을 직접 결정하거든요. 이 섹션을 통해 벡터 데이터베이스의 "왜"와 "어떻게"를 확실히 잡고 가겠습니다.

## 핵심 개념

### 개념 1: 전통적 데이터베이스 vs 벡터 데이터베이스 — 도서관의 두 가지 검색법

> 💡 **비유**: 전통적 데이터베이스는 **도서관의 분류 카탈로그**와 같습니다. "저자: 김영하, 출판년도: 2022"처럼 정확한 조건으로 책을 찾죠. 반면 벡터 데이터베이스는 **사서에게 "이런 느낌의 책 있나요?"**라고 물어보는 것과 같습니다. 정확한 키워드가 아니라, **의미적으로 비슷한 것**을 찾아줍니다.

전통적 관계형 데이터베이스(RDBMS)는 SQL 쿼리로 **정확한 일치(exact match)** 검색에 최적화되어 있습니다. `WHERE name = '홍길동'`처럼 명확한 조건을 줘야 하죠. 하지만 "홍길동과 비슷한 성격의 인물을 찾아줘"라는 질문에는 속수무책입니다.

벡터 데이터베이스는 이 문제를 해결합니다. 데이터를 고차원 벡터(숫자 배열)로 저장하고, 벡터 간 거리를 기준으로 **유사도 검색(Similarity Search)**을 수행합니다.

| 특성 | 전통적 DB (RDBMS) | 벡터 데이터베이스 |
|------|-------------------|------------------|
| 데이터 형태 | 행과 열 (테이블) | 고차원 벡터 + 메타데이터 |
| 검색 방식 | 정확한 조건 매칭 (SQL) | 유사도 검색 |
| 인덱스 | B-Tree, Hash | HNSW, IVF, PQ |
| 대표 쿼리 | `WHERE price > 1000` | "이 벡터와 가장 가까운 10개" |
| 강점 | 트랜잭션, 정합성 | 의미적 검색, 유연성 |
| 약점 | 의미 검색 불가 | ACID 트랜잭션 제한적 |

```run:python
# 전통적 검색 vs 벡터 검색의 차이를 간단히 체감해봅시다
documents = [
    "파이썬으로 웹 크롤링하는 방법",
    "Python 데이터 분석 라이브러리 Pandas 소개",
    "자바스크립트 React 프레임워크 튜토리얼",
    "머신러닝 모델 학습시키기",
    "딥러닝과 신경망의 기초",
]

# 전통적 검색: 키워드가 정확히 일치해야 함
query = "파이썬 데이터 분석"
keyword_results = [doc for doc in documents if "파이썬" in doc and "데이터" in doc]
print(f"키워드 검색 '{query}': {keyword_results}")
# "Python"은 "파이썬"과 같은 뜻이지만 키워드 불일치로 놓침!

# 벡터 검색(유사도 검색)이라면?
# → "파이썬 데이터 분석" 벡터와 가장 가까운 문서를 찾아
#   "Python 데이터 분석 라이브러리 Pandas 소개"도 함께 반환!
print("\n벡터 검색이라면 'Python'과 '파이썬'의 의미적 유사성을 이해하여")
print("더 정확한 검색 결과를 반환합니다.")
```

```output
키워드 검색 '파이썬 데이터 분석': []

벡터 검색이라면 'Python'과 '파이썬'의 의미적 유사성을 이해하여
더 정확한 검색 결과를 반환합니다.
```

위 예제에서 보듯이, "파이썬"이라는 키워드로 검색하면 "Python"이라고 적힌 문서는 놓쳐버립니다. 유사도 검색은 단어의 **의미**를 비교하기 때문에 이런 한계를 극복할 수 있죠.

> ⚠️ **흔한 오해**: "벡터 데이터베이스가 전통적 DB를 대체한다"고 생각하기 쉽지만, 실제 프로덕션에서는 **두 가지를 함께 사용**합니다. 사용자 계정, 결제 정보 등은 관계형 DB에, 문서 검색은 벡터 DB에 맡기는 식이죠. 서로 대체재가 아니라 **보완재**입니다.

### 개념 2: ANN(Approximate Nearest Neighbor) — 완벽보다 빠른 "충분히 좋은" 검색

> 💡 **비유**: 100만 명이 모인 축구장에서 나와 가장 닮은 사람을 찾는다고 상상해보세요. **모든 사람을 한 명씩 비교**(Brute Force)하면 정확하지만, 며칠이 걸립니다. 대신 "안경 쓴 사람들만 먼저 모아보자", "키 170~180cm인 사람들만 보자"처럼 **범위를 좁혀서** 찾으면, 아주 약간 덜 정확하더라도 몇 분 만에 답을 얻을 수 있습니다. 이것이 ANN의 핵심 아이디어입니다.

정확한 최근접 이웃 검색(KNN, K-Nearest Neighbors)은 쿼리 벡터와 **모든 저장된 벡터** 사이의 거리를 계산합니다. 벡터가 $N$개이고 차원이 $d$일 때, 시간 복잡도는 $O(N \times d)$입니다.

$$\text{Brute Force 검색 시간} = O(N \times d)$$

- $N$: 저장된 벡터 수 (예: 1,000,000)
- $d$: 벡터 차원 수 (예: 1,536)

100만 개 × 1,536차원이면 매 쿼리마다 약 15억 번의 곱셈/덧셈이 필요합니다. 실시간 서비스에서는 감당할 수 없는 비용이죠.

ANN은 **정확도를 아주 조금 희생하는 대신(보통 95~99% 수준의 재현율)** 검색 속도를 **수백~수천 배** 향상시킵니다. 실제로 대부분의 RAG 애플리케이션에서 "Top 5 중 1개를 놓치는 것"보다 "응답 시간이 10초 걸리는 것"이 훨씬 치명적이기 때문에, ANN은 사실상 **표준**이 되었습니다.

```run:python
import time

# Brute Force vs ANN의 속도 차이를 시뮬레이션
# (실제 벡터 연산 대신 개념적 비교)

n_vectors = 1_000_000  # 100만 개 벡터
brute_force_ops = n_vectors  # 모든 벡터와 비교
ann_ops = int(n_vectors ** 0.5)  # ANN은 대략 √N 수준으로 줄임

print(f"저장된 벡터 수: {n_vectors:,}개")
print(f"Brute Force 비교 횟수: {brute_force_ops:,}회")
print(f"ANN 비교 횟수 (근사): {ann_ops:,}회")
print(f"속도 향상: 약 {brute_force_ops // ann_ops}배 빠름")
print(f"재현율(Recall): ~95-99% (거의 대부분의 정답을 찾음)")
```

```output
저장된 벡터 수: 1,000,000개
Brute Force 비교 횟수: 1,000,000회
ANN 비교 횟수 (근사): 1,000회
속도 향상: 약 1000배 빠름
재현율(Recall): ~95-99% (거의 대부분의 정답을 찾음)
```

### 개념 3: HNSW — 계층적으로 뛰어다니는 그래프 검색

> 💡 **비유**: HNSW는 **지하철 노선도**와 비슷합니다. 서울에서 부산까지 갈 때, 처음에는 KTX(고속, 정차역 적음 = 상위 레이어)를 타고 대략적인 위치로 이동합니다. 그다음 일반 열차(중간 레이어)로 갈아타고, 마지막에 버스(하위 레이어, 정차역 많음)로 정확한 목적지에 도착하죠. HNSW도 이렇게 **상위에서 하위로 점점 세밀하게** 검색 범위를 좁혀갑니다.

HNSW(Hierarchical Navigable Small World)는 현재 가장 널리 사용되는 ANN 알고리즘입니다. 다층 그래프 구조로 벡터를 연결하고, 상위 레이어에서 하위 레이어로 내려가며 검색 범위를 좁혀갑니다.

**HNSW의 작동 원리:**

```
Layer 2 (최상위):  A ---------> D          ← 적은 노드, 빠른 탐색
                   |             |
Layer 1 (중간):    A --- B ----> D --- E    ← 중간 밀도
                   |    |        |    |
Layer 0 (최하위):  A-B-C-D-E-F-G-H-I-J     ← 모든 노드, 정밀 탐색
```

1. **상위 레이어**(Layer 2)에서 시작: 노드가 적어 빠르게 대략적인 위치를 찾음
2. **중간 레이어**(Layer 1)로 내려감: 더 많은 노드 중 가까운 이웃 탐색
3. **최하위 레이어**(Layer 0)에서 정밀 검색: 모든 노드가 있어 정확한 최근접 이웃 반환

HNSW의 성능을 제어하는 핵심 파라미터는 다음과 같습니다:

| 파라미터 | 의미 | 효과 |
|----------|------|------|
| `M` | 각 노드의 최대 연결 수 | 클수록 정확하지만 메모리 증가 |
| `ef_construction` | 인덱스 구축 시 탐색 범위 | 클수록 정확한 그래프, 구축 시간 증가 |
| `ef_search` | 검색 시 탐색 범위 | 클수록 정확한 검색, 속도 감소 |

> 💡 **ChromaDB와 HNSW**: ChromaDB는 내부적으로 **hnswlib**를 래핑하여 사용하며, 위 파라미터들을 컬렉션 생성 시 `metadata`를 통해 설정할 수 있습니다 (기본값: `M=16`, `ef_construction=100`). 대부분의 경우 기본값으로 충분하며, 파라미터별 세부 튜닝 전략은 Ch7에서 다룹니다.

```python
# ChromaDB에서 HNSW 파라미터를 metadata로 설정하는 예시
client.create_collection(
    "my_collection",
    metadata={
        "hnsw:space": "cosine",
        "hnsw:M": 16,                   # 노드 최대 연결 수
        "hnsw:construction_ef": 100,     # 인덱스 구축 시 탐색 범위
    }
)
```

### 개념 4: IVF — 클러스터로 나눠서 검색 범위 줄이기

> 💡 **비유**: IVF는 **대형 마트의 매장 구역**과 같습니다. 우유를 찾을 때 마트 전체를 돌아다니지 않죠. "유제품 코너"로 직행해서 그 안에서만 찾습니다. IVF도 벡터 공간을 여러 구역(클러스터)으로 나누고, 쿼리와 가장 가까운 구역에서만 검색합니다.

IVF(Inverted File Index)는 k-means 같은 클러스터링 알고리즘으로 벡터 공간을 여러 영역으로 분할합니다.

**IVF의 작동 원리:**

```
전체 벡터 공간
┌──────────┬──────────┬──────────┐
│ 클러스터1 │ 클러스터2 │ 클러스터3 │
│  ● ●     │    ● ●   │  ●       │
│    ●     │  ●       │    ● ●   │
│  ●       │    ● ●   │  ●   ●   │
│     ★    │          │          │  ★ = 중심점(Centroid)
└──────────┴──────────┴──────────┘

쿼리 벡터 → 가장 가까운 중심점 찾기 → 해당 클러스터만 검색
```

1. **인덱스 구축**: 모든 벡터를 k-means로 $k$개 클러스터로 분류
2. **검색 시**: 쿼리 벡터와 가장 가까운 클러스터의 중심점(Centroid)을 찾음
3. **클러스터 내 검색**: 선택된 클러스터 내의 벡터들만 비교

IVF의 핵심 파라미터는 다음과 같습니다. 이 파라미터들은 주로 **FAISS**에서 직접 튜닝하며, ChromaDB는 HNSW를 기본 알고리즘으로 사용하므로 IVF 파라미터를 다루지 않습니다:

| 파라미터 | 의미 | 효과 |
|----------|------|------|
| `nlist` | 전체 클러스터 수 | 클수록 빠르지만 학습 시간 증가 |
| `nprobe` | 검색할 클러스터 수 | 클수록 정확하지만 속도 감소 |

> 📌 **Ch7.1 미리보기**: 여기서 소개한 `nlist`와 `nprobe` 파라미터는 [Ch7.1: FAISS 심화](ch07/01)에서 `IndexIVFFlat`을 직접 구축하며 실습합니다. `nlist` 값에 따라 검색 속도와 정확도가 어떻게 달라지는지 코드로 확인할 수 있으니, 지금은 "클러스터를 나눠서 검색 범위를 줄인다"는 핵심 아이디어만 잡고 가세요. HNSW의 `ef_search`/`ef_construction` 튜닝 역시 Ch7에서 FAISS, Pinecone, Qdrant를 비교하며 함께 다룹니다.

**HNSW vs IVF 비교:**

| 특성 | HNSW | IVF |
|------|------|-----|
| 구조 | 다층 그래프 | 클러스터 파티션 |
| 검색 속도 | 매우 빠름 | 빠름 |
| 메모리 사용 | 높음 (그래프 저장) | 낮음 (중심점만 저장) |
| 정확도 | 매우 높음 | 클러스터 품질에 의존 |
| 인덱스 구축 | 느림 | 비교적 빠름 |
| 대표 사용처 | 대부분의 벡터 DB 기본값 | FAISS, 대규모 데이터 |

### 개념 5: 벡터 데이터베이스 시장 현황 — 누가 누구인가

2024년 약 17.3억 달러 규모였던 벡터 데이터베이스 시장은 2032년까지 약 106억 달러로 성장할 것으로 전망됩니다. RAG와 AI 애플리케이션의 폭발적 성장이 이 시장을 견인하고 있죠.

주요 플레이어를 유형별로 정리해보겠습니다:

| 데이터베이스 | 유형 | 특징 | 최적 용도 |
|-------------|------|------|----------|
| **ChromaDB** | 오픈소스, 임베디드 | 설치 간편, 빠른 프로토타이핑 | 학습, MVP, 소규모 프로젝트 |
| **FAISS** | 오픈소스, 라이브러리 | Meta 개발, 최고 성능의 벡터 검색 엔진 | 연구, 대규모 배치 처리 |
| **Pinecone** | 클라우드 매니지드 | 완전 관리형, 자동 스케일링 | 프로덕션, 엔터프라이즈 |
| **Qdrant** | 오픈소스/클라우드 | Rust 기반, 강력한 메타데이터 필터링 | 복합 필터링이 필요한 서비스 |
| **Milvus** | 오픈소스/클라우드 | 대규모 분산 시스템, GPU 지원 | 초대규모 데이터셋 |
| **Weaviate** | 오픈소스/클라우드 | 모듈형 아키텍처, 하이브리드 검색 | 멀티모달 검색 |

> 🔥 **실무 팁**: 벡터 DB 선택에 정답은 없습니다. **프로토타이핑 단계**에서는 ChromaDB로 빠르게 시작하고, **프로덕션**으로 갈 때 Pinecone이나 Qdrant로 마이그레이션하는 전략이 실무에서 흔합니다. 이 코스에서는 Ch6에서 ChromaDB를, Ch7에서 FAISS/Pinecone/Qdrant를 다룹니다.

## 실습: 직접 해보기

벡터 데이터베이스의 핵심 동작을 ChromaDB로 직접 체험해봅시다. ChromaDB는 `pip install chromadb` 한 줄이면 바로 시작할 수 있어 학습용으로 최적입니다.

```python
# 필요한 패키지 설치
# pip install chromadb
```

```run:python
import chromadb

# 1. 클라이언트 생성 (인메모리 모드 — 설치 즉시 사용 가능)
# EphemeralClient: 메모리에만 데이터를 저장하는 모드
# 6.2에서 PersistentClient, HttpClient 등 공식 클라이언트 모드를 학습합니다
client = chromadb.EphemeralClient()

# 2. 컬렉션 생성 (= 전통 DB의 테이블에 해당)
collection = client.create_collection(
    name="my_documents",
    metadata={"hnsw:space": "cosine"}  # 코사인 유사도 사용
)

# 3. 문서 추가 — ChromaDB가 자동으로 임베딩 생성!
collection.add(
    documents=[
        "파이썬은 데이터 분석에 널리 사용되는 프로그래밍 언어입니다",
        "자바스크립트는 웹 개발의 핵심 언어입니다",
        "머신러닝은 데이터에서 패턴을 학습하는 AI 기술입니다",
        "딥러닝은 신경망을 활용한 머신러닝의 한 분야입니다",
        "React는 페이스북이 만든 프론트엔드 라이브러리입니다",
    ],
    ids=["doc1", "doc2", "doc3", "doc4", "doc5"],  # 고유 ID 필수
    metadatas=[  # 메타데이터로 필터링 가능
        {"category": "programming", "lang": "python"},
        {"category": "programming", "lang": "javascript"},
        {"category": "ai", "lang": "python"},
        {"category": "ai", "lang": "python"},
        {"category": "programming", "lang": "javascript"},
    ]
)

print(f"컬렉션 '{collection.name}'에 {collection.count()}개 문서 저장 완료!")

# 4. 유사도 검색 — "데이터 분석 도구"와 가장 비슷한 문서 3개 찾기
results = collection.query(
    query_texts=["데이터 분석에 좋은 도구는?"],
    n_results=3  # 상위 3개 반환
)

print("\n🔍 쿼리: '데이터 분석에 좋은 도구는?'")
print("=" * 50)
for i, (doc, distance) in enumerate(
    zip(results["documents"][0], results["distances"][0])
):
    print(f"  {i+1}. (거리: {distance:.4f}) {doc}")

# 5. 메타데이터 필터링 결합 — AI 카테고리에서만 검색
filtered_results = collection.query(
    query_texts=["인공지능 기술"],
    n_results=2,
    where={"category": "ai"}  # AI 카테고리만 필터링
)

print("\n🔍 쿼리: '인공지능 기술' (category='ai' 필터)")
print("=" * 50)
for i, doc in enumerate(filtered_results["documents"][0]):
    print(f"  {i+1}. {doc}")
```

```output
컬렉션 'my_documents'에 5개 문서 저장 완료!

🔍 쿼리: '데이터 분석에 좋은 도구는?'
==================================================
  1. (거리: 0.5765) 파이썬은 데이터 분석에 널리 사용되는 프로그래밍 언어입니다
  2. (거리: 1.0432) 머신러닝은 데이터에서 패턴을 학습하는 AI 기술입니다
  3. (거리: 1.2187) 딥러닝은 신경망을 활용한 머신러닝의 한 분야입니다

🔍 쿼리: '인공지능 기술' (category='ai' 필터)
==================================================
  1. 머신러닝은 데이터에서 패턴을 학습하는 AI 기술입니다
  2. 딥러닝은 신경망을 활용한 머신러닝의 한 분야입니다
```

이 코드에서 주목할 점이 몇 가지 있습니다:

1. **자동 임베딩**: 문서 텍스트만 넣으면 ChromaDB가 알아서 벡터로 변환합니다 (기본: Sentence Transformers 모델)
2. **유사도 검색**: "데이터 분석"이라는 쿼리에 "파이썬 데이터 분석" 문서가 가장 가까운 결과로 반환됩니다
3. **메타데이터 필터링**: `where` 조건으로 유사도 검색과 전통적 필터링을 결합할 수 있습니다
4. **거리(distance)**: 값이 작을수록 의미적으로 더 유사합니다

## 더 깊이 알아보기

### HNSW의 탄생 — 스몰 월드 현상에서 검색 알고리즘까지

HNSW 알고리즘에는 흥미로운 탄생 스토리가 있습니다. 이 알고리즘의 이름에 있는 "Small World"는 사회학에서 유래합니다.

1967년, 사회심리학자 **스탠리 밀그램(Stanley Milgram)**이 유명한 "6단계 분리(Six Degrees of Separation)" 실험을 수행했습니다. 미국 네브래스카주의 무작위 사람에게 편지를 건네고, 아는 사람을 통해 보스턴의 특정 인물에게 전달하도록 요청했죠. 놀랍게도 평균 **5.5단계**만 거치면 편지가 도착했습니다. 60억 인구의 세상이 생각보다 "좁다(small)"는 것이죠.

이 아이디어가 그래프 이론으로 발전하여 "Navigable Small World" 그래프가 탄생했고, 2012년 러시아의 연구자 **유리 말코프(Yury Malkov)**가 이를 벡터 검색에 적용했습니다. 2016년에는 말코프와 **드미트리 야슈닌(Dmitry Yashunin)**이 여기에 **계층 구조(Hierarchical)**와 **스킵 리스트(Skip List)** 아이디어를 결합하여 HNSW를 발표했습니다([논문: arXiv:1603.09320](https://arxiv.org/abs/1603.09320)).

> 💡 **알고 계셨나요?**: HNSW의 "H"(Hierarchical)는 스킵 리스트라는 자료구조에서 영감을 받았습니다. 스킵 리스트는 1989년 윌리엄 퓨(William Pugh)가 발명한 것으로, 연결 리스트에 여러 층의 "고속도로"를 추가해 검색을 빠르게 만드는 기법입니다. 말코프는 이 아이디어를 Small World 그래프에 접목하여, **다층 그래프에서 위에서 아래로 내려가며 검색**하는 HNSW를 완성했습니다.

오늘날 HNSW는 사실상 **모든 주요 벡터 데이터베이스의 기본 인덱싱 알고리즘**이 되었습니다. ChromaDB, Qdrant, Weaviate, Pinecone 등 거의 모든 벡터 DB가 HNSW를 Rust, Go, C++, Java 등 다양한 언어로 구현하여 사용하고 있죠.

### 벡터 데이터베이스 붐의 시작

벡터 데이터베이스라는 개념 자체는 새로운 것이 아닙니다. FAISS는 2017년에 Meta(당시 Facebook)가 공개했고, 그 전부터 연구 커뮤니티에서 ANN 검색은 활발히 연구되었죠. 하지만 2022년 말 ChatGPT의 등장과 함께 RAG 패턴이 폭발적으로 인기를 얻으면서, 벡터 데이터베이스 시장이 본격적으로 열렸습니다. Pinecone, Weaviate, Qdrant 등이 연이어 대규모 투자를 유치했고, 이 시장은 2024년 약 17억 달러에서 2032년 106억 달러로 성장할 것으로 예측됩니다.

## 흔한 오해와 팁

> ⚠️ **흔한 오해**: "벡터 데이터베이스는 AI/ML 전용이다"라고 생각하기 쉽지만, 추천 시스템, 이상 탐지, 이미지/음악 검색, 중복 감지 등 **유사성을 기반으로 하는 모든 애플리케이션**에서 활용됩니다. RAG는 벡터 DB의 여러 활용 사례 중 하나일 뿐입니다.

> 💡 **알고 계셨나요?**: PostgreSQL의 **pgvector** 확장을 쓰면 기존 관계형 DB에 벡터 검색 기능을 추가할 수 있습니다. 최근 벤치마크에서 pgvectorscale은 5,000만 벡터에서 99% 재현율 기준 471 QPS를 달성하여, 일부 전용 벡터 DB를 능가하는 성능을 보여주기도 했습니다. "반드시 전용 벡터 DB를 써야 하나?"라는 질문에 대한 답은 프로젝트의 규모와 요구사항에 따라 달라집니다.

> 🔥 **실무 팁**: ChromaDB에서 HNSW의 `ef_search` 파라미터는 컬렉션 생성 시 `metadata`로 설정할 수 있습니다: `metadata={"hnsw:search_ef": 100}`. 개발 중에는 낮은 값으로 빠른 반복을, 정확도가 중요한 최종 평가에서는 높은 값을 설정하는 전략이 유용합니다. 이처럼 ChromaDB는 hnswlib의 파라미터를 `metadata`를 통해 간접적으로 노출하며, 각 파라미터의 최적화 전략은 Ch7에서 자세히 다룹니다.

## 핵심 정리

| 개념 | 설명 |
|------|------|
| 벡터 데이터베이스 | 고차원 벡터를 저장하고 유사도 검색을 수행하는 특수 DB |
| 전통 DB와의 차이 | 정확한 조건 매칭(SQL) 대신 벡터 거리 기반의 유사도 검색 수행 |
| ANN | 정확도를 약간 희생(95~99%)하여 검색 속도를 수백~수천 배 향상시키는 근사 검색 |
| HNSW | 다층 그래프 기반 ANN. 상위→하위 레이어로 범위를 좁히며 검색. ChromaDB는 hnswlib 래핑 |
| IVF | 클러스터 기반 ANN. 벡터 공간을 구역으로 나누고 가까운 구역만 검색. 주로 FAISS에서 활용 |
| ChromaDB | 오픈소스, 임베디드 벡터 DB. 설치 간편하여 학습/프로토타이핑에 최적 |
| 메타데이터 필터링 | 유사도 검색과 전통적 조건 필터링을 결합하는 기능 |
